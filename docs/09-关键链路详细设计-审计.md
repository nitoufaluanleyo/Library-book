# 《关键链路详细设计》审计报告

## A. 总体判断

**审计结论：有条件通过。**

文档优先沿用了前序已确认的用户预约闭环和管理预约闭环，没有改动 U-P01 至 U-P05、A-P01、A-P02、A-P04 的页面名称和核心职责。用户链路覆盖账号上下文、选座、预约、查看、取消和重新预约；管理链路覆盖登录、统计、筛选、取消、统计回流和退出，两条链路均能闭环。

Controller、Service、DataAccess 的分工基本合理，数据库变化与 4 表设计、Reserved/Cancelled 存储状态、动态 Completed、过滤唯一索引和 RowVersion 保持一致。异常、权限和并发边界已经达到进入代码实现的粒度。

主要待确认项是 A-P02 的筛选范围：页面树和系统设计只明确日期、状态筛选，数据库设计和本详细设计又加入区域、体验用户筛选。该差异必须在 Sprint 0 前统一，否则会出现页面原型、Controller 参数和验收标准不一致。

## B. 发现的问题清单

| 编号 | 严重程度 | 发现的问题 | 审计判断 |
| --- | --- | --- | --- |
| B-01 | 中 | A-P02 筛选范围与早期页面口径不完全一致 | `docs/04` 和 `docs/07` 只明确日期、状态；`docs/08` 要求数据库支持区域、用户查询；`docs/09` 直接把四类条件都放入页面路由和 Controller。数据库具备查询能力不必然等于本期页面增加两个筛选控件。 |
| B-02 | 中 | 动态展示状态存在两套表达形式 | 文档同时给出 `GetDisplayStatus` 的内存规则和 A-P02 的 EF Core 查询条件。两者逻辑一致，但若代码分别手写，后续容易出现“页面显示已完成、筛选却查不到”的漂移。 |
| B-03 | 中 | 管理员密码校验方法只写到 `ValidateCredentialsAsync` | 前序已要求 PasswordHash，但详细设计未固定哈希验证方式。进入代码前需明确使用安全哈希验证，不得退化为明文比较或自行做简单 SHA 哈希。 |
| B-04 | 低 | U-P04 重组页面数据的失败分支还不够具体 | ModelState 或业务校验失败后会调用 `BuildCreatePageAsync`，但如果此时座位已被删除或停用，应明确跳回 U-P03/U-P02，而不是再次渲染不可用表单。 |
| B-05 | 低 | 唯一索引异常按索引名称转换依赖 SQL Server 异常解析 | 当前方案可行，但实现时需要从内部 `SqlException` 检查 2601/2627 和索引名，不能对全部 `DbUpdateException` 做字符串模糊匹配。 |
| B-06 | 低 | Service 方法契约中的 CancellationToken 写法不完全统一 | Controller 顺序中包含 cancellationToken，方法契约清单部分省略。实现时应统一传递到 EF Core 异步调用。 |
| B-07 | 低 | 管理员取消接收 AdminId 但不持久化操作者编号 | 文档已经明确 AdminId 只用于认证和日志上下文，数据库只保存 `CancelledBy = Admin`；当前单管理员范围下合理。 |
| B-08 | 低 | 路由与页面树早期建议路径不同 | 文档已明确以较新的系统设计路由为准，页面编号和职责未变，因此不构成擅自改链路。 |

### 已通过检查

1. 用户主链路沿用 U-P01 → U-P02 → U-P03 → U-P04 → U-P05。
2. 用户取消后停留 U-P05，并能回到选座链路重新预约。
3. 管理链路沿用 A-P01 → A-P04 → A-P02 → A-P04 → A-P01。
4. GET/POST 路由、输入参数和成功跳转均清楚，写操作采用 PRG。
5. 当前体验用户由 Controller 从 Session 解析后显式传给业务 Service，Service 不直接依赖 HttpContext。
6. Controller 不写冲突规则，Service 不负责页面跳转，Razor 不访问 DbContext。
7. DataAccess 不新增 Repository，EF Core 查询和 Configuration 边界与系统设计一致。
8. 创建预约和两类取消都有明确校验顺序、错误代码和数据库变化。
9. 两个过滤唯一索引使用正确字段 `ReservationDate`、`ExperienceUserId`。
10. RowVersion、CancelledAt、CancelledBy 和统计变化与数据库设计对应。
11. Reserved、Cancelled、Completed 和两类取消时间边界与 `docs/08` 一致。
12. 座位停用、非法日期、过期时段、两类冲突、越权取消、重复提交、重复取消和并发取消均已覆盖。
13. “默认补足链路 / 当前假设”已显式说明，且没有新增第三条主链路或额外页面。

## C. 必改项

以下事项必须在 Sprint 0 任务拆分或代码开始前确定：

1. **冻结 A-P02 筛选范围。** 保守方案是页面继续只显示日期、状态筛选，区域和用户仅保留为 Service/DataAccess 查询能力；若确定页面也展示四类筛选，必须同步更新页面验收口径，不能只改 Controller。
2. **统一动态状态规则。** 建立一个 `ReservationStatusRules` 或等价规则组件，集中保存时段起止时间和状态判定；数据库查询表达式与内存计算必须共用同一组常量并用测试矩阵校验。
3. **固定密码哈希验证。** Sprint 0 明确管理员密码哈希生成和验证方案，种子数据只保存哈希，登录失败始终返回统一提示。
4. **固定 U-P04 页面重组失败去向。** Seat 不存在时返回 U-P02，Seat 已停用时返回 U-P03 并提示；仅字段或冲突错误留在 U-P04 回显。
5. **固定数据库异常识别。** 只把 SQL Server 2601/2627 且命中两个约定过滤索引的异常转换为 SeatConflict/UserConflict，其他异常继续上抛。

## D. 建议优化项

1. 为每个 Service 方法补齐输入、返回类型和 CancellationToken，例如 `Task<ServiceResult<T>>`。
2. 将 `CanUserCancel`、`CanAdminCancel` 与 DisplayStatus 放进同一状态规则测试表，覆盖开始前、进行中、结束后三个时间点。
3. 为 A-P02 筛选条件建立独立 `AdminReservationFilterViewModel`，取消后用明确路由值恢复条件。
4. 为创建预约的两个唯一索引异常各写一个集成测试，不只测试 Service 预查询。
5. 管理员登录使用框架提供的密码哈希器或标准 PBKDF2 实现，不自行设计密码算法。
6. 将 DbSeeder 的“核心表全空才初始化”写成一个明确判断，并测试重复执行不增加数据。
7. Sprint 0 只先创建链路需要的最小 Partial 和样式，不提前实现未进入主链路的装饰组件。

## E. 是否可以进入下一步“开发准备与 Sprint 0”

**可以有条件进入。**

关键链路已经足以指导 Controller、Service、EF Core Configuration、Migration 和首批测试开发，不需要重新设计流程。Sprint 0 开始前应先完成 C 节五项口径确认，尤其是 A-P02 筛选范围和动态状态单一规则源。

确认后可按文档第 10.2 节顺序推进：先完成建库与种子数据，再实现体验用户和座位查询、用户预约与取消、管理员登录与取消、五项统计，最后补座位管理。不得在开发阶段新增页面、实体或第三套状态。
