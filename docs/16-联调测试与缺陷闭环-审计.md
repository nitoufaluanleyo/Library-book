## B. 发现的问题清单

### B1. 通过项：联调和测试场景来自既有承诺

docs/16 覆盖了以下既有场景：

- 体验账号切换、查座、预约、回看和用户取消；
- 管理员登录、后台保护、预约筛选和处理、座位维护、统计和退出；
- 两类预约冲突、停用座位、非法取消、并发变化和状态回流；
- Reserved、Cancelled、Completed 统一口径；
- 空状态、Edge/Chrome、390px 用户端和 1366px 管理端。

这些场景能追溯到成功标准、任务板和前序记录，没有临时扩大项目范围。

### B2. P0：用户端与管理端没有联通

实际检查结果：

- `LibrarySeatReservation.sln` 不存在；
- `src/` 不存在；
- Web `.csproj` 不存在；
- EF Core、LocalDB、Migration 和种子数据不存在。

因此用户预约无法写入，管理端无法查询或处理同一条数据，用户端也无法重新查询管理结果。状态回流没有实现证据。

### B3. P0：主流程不能完整演示

TC-01 至 TC-14 全部处于阻塞状态。当前只能打开静态原型，不能验证 Session、Cookie、数据库、权限、冲突、取消或统计。静态页面跳转不能作为完整主流程验收。

### B4. P0：关键规则没有生效证据

当前没有 Service、Controller、数据库约束或可执行测试，以下规则均未验证：

- SeatConflict 和 UserConflict；
- 停用座位不可预约；
- 越权、开始后、重复和并发取消；
- 管理员取消后的状态、时段和统计回流；
- 用户端与管理端统一动态状态。

## C. 必改项

1. **先关闭系统测试目标 P0。** 完成开发前门禁、工程骨架、数据库、用户端和管理端主链路，使应用可重复 build、run 和重建数据。
2. **建立 Playwright Level B 骨架。** 创建 package，安装 `@playwright/test`，补 `playwright.config.*`，实际配置 `channel: 'msedge'`、baseURL、webServer、reuseExistingServer 和合理超时。
3. **执行真实自动点击。** 至少完成“查看座位 -> 提交预约 -> 我的预约”和“管理员登录 -> 处理预约 -> 用户端回看”中的 1 至 2 条链路。
4. **补最小 smoke 脚本。** 自动检查应用启动、首页、用户端入口和管理端登录入口，并记录命令、退出码和失败点。
5. **执行兼容性和回归。** Edge、Chrome、390px 用户端、1366px 管理端均需有实际页面结果；每次修复后重跑主流程。
6. **形成至少一个真实 Bug 闭环。** 记录发现、修复提交、复测命令和关闭证据，不能只登记未实现项。
7. **同步任务板缺陷卡。** 将 BUG-16-001 至 BUG-16-007 或后续真实缺陷逐条写入任务板，记录状态、关联 T16 卡片和关闭证据。
8. **P0 清零后再申请交付。** 阶段最低完成线和 T16 验收未通过前，不得关闭 Sprint 4 或 M4。

## D. 建议优化项

1. 将“阶段阻塞项”和“运行中软件缺陷”分开统计，避免把尚未开发与实现错误混为同一种 Bug。
2. Playwright 用例优先复用种子数据，并在每轮前重置数据库，降低顺序依赖。
3. 自动化失败时保留 trace、截图和控制台日志，方便课堂展示定位过程。
4. smoke 脚本应返回非零退出码表示失败，便于后续接入 CI 或发布前检查。
5. Node.js 24 当前可用；安装 Playwright 后应记录实际依赖版本和兼容结果，不提前假定全部兼容。
