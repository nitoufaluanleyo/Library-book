## 0. 前序文档识别结果

### 0.1 已确定关键链路

前序文档已经明确两条主链路，本设计不改名、不改页面职责：

1. **用户预约闭环：** U-P01 用户首页 → U-P02 座位列表 → U-P03 座位详情 → U-P04 预约提交 → U-P05 我的预约，并包含用户取消后重新预约的支链路。
2. **管理预约与状态回流闭环：** A-P01 管理员登录 → A-P04 统计 → A-P02 预约管理筛选/取消 → A-P04 查看统计变化 → A-P01 退出。

### 0.2 已确定页面动作

| 页面 | 已确定动作 |
| --- | --- |
| U-P01 | 查看座位、进入我的预约、切换体验账号 |
| U-P02 | 按区域筛选、清除筛选、查看座位详情 |
| U-P03 | 查看座位信息；启用座位进入预约提交 |
| U-P04 | 选择 7 天范围内日期、选择固定时段、提交预约 |
| U-P05 | 查看当前体验用户预约、取消尚未开始预约、返回座位详情 |
| A-P01 | 管理员登录 |
| A-P02 | 按日期、展示状态筛选；取消尚未结束预约。区域、用户条件仅保留在 Service/DataAccess 查询能力中 |
| A-P04 | 查看 5 个统计指标、进入预约管理、退出 |

### 0.3 已确定权限边界

1. 当前体验用户编号只从 Session `CurrentExperienceUserId` 读取，不接受表单中的 UserId。
2. 用户只能查看和取消自己的预约。
3. 管理端使用 `AdminCookie`，未登录访问 A-P02、A-P03、A-P04 时跳转 A-P01。
4. 用户 Session 与管理员 Cookie 相互独立。
5. 所有 POST 请求使用防伪令牌，所有写操作在 Service 中重新校验数据库状态。

### 0.4 已确定状态与异常

1. 预约数据库只保存 `Reserved = 1`、`Cancelled = 2`。
2. `Completed` 只作为动态展示状态，不写入数据库。
3. 学生只能在预约开始前取消；管理员可以在预约结束前取消。
4. 座位停用后禁止新预约，但不自动取消已有预约。
5. 创建预约必须拦截座位冲突和用户冲突。
6. 重复提交、重复取消、越权取消、并发状态变化均不能产生错误数据。

## 1. 关键链路选择说明

### 1.1 链路一：用户预约与用户取消闭环

选择原因：该链路覆盖体验用户上下文、座位查询、日期时段、两类冲突、预约写入、动态状态、用户数据隔离和取消释放，是本项目最核心的业务链路。

链路范围：

```text
U-P01 确认/切换体验账号
  -> U-P02 筛选座位
  -> U-P03 查看座位详情
  -> U-P04 提交预约
  -> U-P05 查看预约
  -> U-P05 用户取消
  -> U-P02/U-P03/U-P04 可重新预约
```

### 1.2 链路二：管理预约与状态回流闭环

选择原因：该链路覆盖管理员 Cookie、后台访问保护、动态展示状态筛选、管理员取消、RowVersion 并发处理和统计回流，是管理端最重要的处理闭环。

链路范围：

```text
A-P01 管理员登录
  -> A-P04 查看统计
  -> A-P02 按日期/展示状态筛选预约
  -> A-P02 管理员取消
  -> A-P04 重新查询统计
  -> A-P01 退出
```

管理座位闭环已经在前序文档中确认，但不属于本轮两条关键链路的展开重点。其代码衔接点在第 10 节保留，不新增第三条主链路。

## 2. 链路一：页面与路由

系统设计阶段已经固定工程路由。本节使用后续系统设计路由，页面编号和职责仍沿用页面树。

| 步骤 | 页面 | HTTP | 路由 | 输入 | 成功去向 |
| --- | --- | --- | --- | --- | --- |
| 1 | U-P01 用户首页 | GET | `/` | 无 | 留在 U-P01 |
| 2 | U-P01 切换体验账号 | POST | `/experience-account/switch` | `experienceUserId` | Redirect `/` |
| 3 | U-P02 座位列表 | GET | `/seats?area={area}` | 可选区域 | U-P02 |
| 4 | U-P03 座位详情 | GET | `/seats/{id}` | SeatId | U-P03 |
| 5 | U-P04 预约提交 | GET | `/reservations/create?seatId={id}&date={date}&slot={slot}` | SeatId，可选日期/时段 | U-P04 |
| 6 | U-P04 提交预约 | POST | `/reservations/create` | SeatId、ReservationDate、TimeSlot | Redirect `/my-reservations` |
| 7 | U-P05 我的预约 | GET | `/my-reservations` | 无 | U-P05 |
| 8 | U-P05 用户取消 | POST | `/my-reservations/{id}/cancel` | ReservationId | Redirect `/my-reservations` |

### 2.1 页面输入安全边界

1. `ReservationCreateViewModel` 只包含 SeatId、ReservationDate、TimeSlot，不包含 ExperienceUserId。
2. 取消表单只提交 ReservationId，不提交 StoredStatus、CancelledBy 和用户编号。
3. GET 查询参数只用于页面选择和回显，POST 时必须重新查询 Seat、ExperienceUser、Reservation。
4. 切换体验账号成功后统一返回 U-P01，不回到正在填写的 U-P04。
5. U-P04 GET 收到不存在或停用的 SeatId 时，跳转 U-P03 或 U-P02 并显示原因，不渲染可提交表单。

## 3. 链路一：Controller 动作

### 3.1 动作清单

| Controller.Action | 主要职责 | 调用 Service | 返回 |
| --- | --- | --- | --- |
| `HomeController.Index()` | 取得当前体验用户和座位概况 | CurrentExperienceUserService、SeatService | `HomeIndexViewModel` |
| `ExperienceAccountController.Switch(int experienceUserId)` | 校验目标用户并更新 Session | CurrentExperienceUserService | Redirect U-P01 |
| `SeatsController.Index(string? area)` | 规范化区域筛选并查询座位 | SeatService | `SeatListViewModel` |
| `SeatsController.Details(int id)` | 查询座位详情 | SeatService | `SeatDetailsViewModel` / NotFound |
| `ReservationsController.Create(int seatId, DateOnly? date, ReservationTimeSlot? slot)` GET | 解析当前用户并生成日期、时段及两类占用状态 | CurrentExperienceUserService、ReservationService | `ReservationCreateViewModel` |
| `ReservationsController.Create(ReservationCreateViewModel model)` POST | 解析当前用户、调用创建服务、处理结果 | CurrentExperienceUserService、ReservationService | View 或 Redirect U-P05 |
| `ReservationsController.Mine()` | 解析当前用户并查询自己的预约 | CurrentExperienceUserService、ReservationService | `MyReservationsViewModel` |
| `ReservationsController.Cancel(int id)` POST | 解析当前用户并调用用户取消 | CurrentExperienceUserService、ReservationService | Redirect U-P05 |

U-P04 GET 先由 CurrentExperienceUserService 取得 currentExperienceUserId，再调用 `BuildCreatePageAsync(currentExperienceUserId, seatId, date, slot)`。页面构建同时查询座位占用和当前用户时段占用，不能只显示 SeatConflict 而遗漏 UserConflict 状态。

### 3.2 创建预约 POST 动作顺序

```text
1. ASP.NET Core 完成模型绑定和防伪令牌校验。
2. ModelState 无效：重新组装 U-P04 页面数据并返回 View。
3. CurrentExperienceUserService 从 Session 读取并校验当前用户。
4. Controller 取得 currentExperienceUserId。
5. 调用 ReservationService.CreateAsync(currentExperienceUserId, model, cancellationToken)。
6. ServiceResult 失败：将错误加入 ModelState，重新组装 U-P04 并返回。
7. ServiceResult 成功：TempData 写入“预约成功”。
8. RedirectToAction(Mine)，执行 POST-Redirect-GET。
```

Controller 不把 Session、HttpContext 或 UserId 传入 ViewModel，也不自行判断冲突。

### 3.3 用户取消 POST 动作顺序

```text
1. 校验防伪令牌。
2. 从 Session 解析 currentExperienceUserId。
3. 调用 ReservationService.CancelByUserAsync(id, currentExperienceUserId, cancellationToken)。
4. 成功：TempData 写入“预约已取消”。
5. 失败：TempData 写入统一安全提示。
6. RedirectToAction(Mine)。
```

取消失败仍重定向 U-P05，避免刷新重复提交。预约不存在和不属于当前用户使用同一个对外错误，不泄露其他用户预约是否存在。

## 4. 链路一：Service 职责

### 4.1 服务方法契约

```text
CurrentExperienceUserService.GetRequiredUserIdAsync()
CurrentExperienceUserService.SwitchAsync(experienceUserId)
SeatService.GetListAsync(area)
SeatService.GetDetailsAsync(seatId)
ReservationService.BuildCreatePageAsync(currentExperienceUserId, seatId, selectedDate, selectedSlot)
ReservationService.CreateAsync(currentExperienceUserId, ReservationCreateViewModel)
ReservationService.GetMineAsync(currentExperienceUserId)
ReservationService.CancelByUserAsync(reservationId, currentExperienceUserId)
ReservationService.GetDisplayStatus(storedStatus, reservationDate, timeSlot, now)
```

业务 Service 接收明确的 `currentExperienceUserId`，不直接读取 HttpContext 或 Session。

### 4.2 CreateAsync 校验与写入顺序

每次操作只读取一次 `TimeProvider.GetLocalNow()`，后续校验使用同一个 `now`。

| 顺序 | 检查 | 失败错误代码 | 是否写库 |
| --- | --- | --- | --- |
| 1 | 体验用户存在且 IsEnabled = true | `ExperienceUserUnavailable` | 否 |
| 2 | Seat 存在 | `SeatNotFound` | 否 |
| 3 | Seat.IsEnabled = true | `SeatDisabled` | 否 |
| 4 | ReservationDate 在 today 至 today + 6 天 | `DateOutOfRange` | 否 |
| 5 | TimeSlot 为 Morning/Afternoon/Evening | `InvalidTimeSlot` | 否 |
| 6 | 若为今天，now 早于该时段开始时间 | `TimeSlotStarted` | 否 |
| 7 | 同 SeatId、ReservationDate、TimeSlot 无 Reserved | `SeatConflict` | 否 |
| 8 | 同 ExperienceUserId、ReservationDate、TimeSlot 无 Reserved | `UserConflict` | 否 |
| 9 | 创建 Reservation，StoredStatus = Reserved | 无 | 是 |
| 10 | SaveChangesAsync | 唯一索引异常转冲突错误 | 成功或回滚 |

### 4.3 冲突检查规则

```text
SeatConflict:
  SeatId == input.SeatId
  AND ReservationDate == input.ReservationDate
  AND TimeSlot == input.TimeSlot
  AND StoredStatus == Reserved

UserConflict:
  ExperienceUserId == currentExperienceUserId
  AND ReservationDate == input.ReservationDate
  AND TimeSlot == input.TimeSlot
  AND StoredStatus == Reserved
```

Service 预检查用于返回清楚提示。数据库过滤唯一索引用于处理两个请求同时通过预检查的并发情况。

### 4.4 CancelByUserAsync 校验顺序

| 顺序 | 检查 | 失败错误代码 |
| --- | --- | --- |
| 1 | Reservation 存在且 ExperienceUserId 等于当前用户 | `ReservationNotFoundOrNotOwned` |
| 2 | StoredStatus = Reserved | `ReservationAlreadyCancelled` |
| 3 | now 早于 ReservationDate + TimeSlot.StartTime | `ReservationAlreadyStarted` |
| 4 | 更新 StoredStatus、CancelledAt、CancelledBy | 无 |
| 5 | SaveChanges 时 RowVersion 未冲突 | `ReservationChanged` |

成功更新：

```text
StoredStatus = Cancelled
CancelledAt = now
CancelledBy = Student
```

### 4.5 展示状态统一计算

```text
StoredStatus == Cancelled
  -> DisplayStatus = Cancelled

StoredStatus == Reserved
AND now >= ReservationDate + TimeSlot.EndTime
  -> DisplayStatus = Completed

其他 Reserved
  -> DisplayStatus = Reserved
```

U-P05 和 A-P02 必须调用同一规则，不允许 Razor 根据文字或 CSS 自行推断状态。

取消按钮不能只看展示状态，必须单独计算：

```text
CanUserCancel = StoredStatus == Reserved AND now < StartDateTime
CanAdminCancel = StoredStatus == Reserved AND now < EndDateTime
```

预约开始后、结束前仍展示“已预约”，但 `CanUserCancel = false`、`CanAdminCancel = true`。

## 5. 链路一：DataAccess 动作

本项目不增加 Repository。ReservationService、SeatService 使用注入的 `AppDbContext` 编写 LINQ；DataAccess 目录负责 DbContext、Entity Configuration、Migration 和 Seed。

### 5.1 页面查询

| 页面 | DbSet 与条件 | 跟踪方式 |
| --- | --- | --- |
| U-P01 | Seats Count、Enabled Count；ExperienceUsers 按 Id 查询 | `AsNoTracking()` |
| U-P02 | Seats，按 Area 可选过滤，SeatNumber 升序 | `AsNoTracking()` |
| U-P03 | Seats 按主键查询 | `AsNoTracking()` |
| U-P04 GET | Seat + 当前 ExperienceUserId + 指定日期 Reserved Reservations，分别投影座位占用和用户占用的 TimeSlot | `AsNoTracking()` |
| U-P05 | Reservations 按 ExperienceUserId，Include/投影 Seat | `AsNoTracking()` |

### 5.2 创建预约写入

1. 使用 `AnyAsync()` 分别执行 SeatConflict、UserConflict 预检查。
2. 构造 Reservation：ExperienceUserId、SeatId、ReservationDate、TimeSlot、StoredStatus、CreatedAt。
3. `DbSet<Reservation>.Add()` 后执行一次 `SaveChangesAsync()`。
4. 单条新增使用 EF Core 默认事务，不手写长事务和锁表。
5. 捕获 SQL Server 唯一冲突错误 2601/2627，并只在命中过滤索引名称时转换：

| 索引 | 转换错误 |
| --- | --- |
| `UX_Reservations_Seat_Date_TimeSlot_Reserved` | `SeatConflict` |
| `UX_Reservations_User_Date_TimeSlot_Reserved` | `UserConflict` |

其他 `DbUpdateException` 继续作为非预期异常交给统一异常处理，不统一伪装成预约冲突。

### 5.3 用户取消写入

1. 以 ReservationId 和 ExperienceUserId 查询跟踪实体。
2. Service 在实体当前状态上完成归属、状态和开始时间复核。
3. 同一次 SaveChanges 更新 StoredStatus、CancelledAt、CancelledBy。
4. RowVersion 由 SQL Server 自动变化。
5. `DbUpdateConcurrencyException` 转为 `ReservationChanged`。

取消 POST 不要求浏览器提交 RowVersion。Service 查询跟踪实体时取得当前 RowVersion；并发请求若同时读取同一版本，后保存请求会发生并发异常。

### 5.4 EF Core 关键配置

枚举固定映射为 `tinyint`，数值不能随意调整：

```text
TimeSlot: Morning = 1, Afternoon = 2, Evening = 3
StoredStatus: Reserved = 1, Cancelled = 2
CancelledBy: Student = 1, Admin = 2
```

过滤唯一索引配置必须生成以下实际列和 Filter：

```text
(SeatId, ReservationDate, TimeSlot) UNIQUE WHERE [StoredStatus] = 1
(ExperienceUserId, ReservationDate, TimeSlot) UNIQUE WHERE [StoredStatus] = 1
```

字段只使用 `ReservationDate`、`ExperienceUserId`，不使用不存在的 `Date`、`UserId`。

## 6. 链路一：数据库变化

### 6.1 创建预约前后

创建前：

```text
Reservations 中不存在：
  同 SeatId + ReservationDate + TimeSlot 的 Reserved
  同 ExperienceUserId + ReservationDate + TimeSlot 的 Reserved
```

创建成功新增一行：

| 字段 | 写入值 |
| --- | --- |
| ExperienceUserId | Controller 从 Session 解析后显式传入 |
| SeatId | U-P04 提交值，Service 已重新验证 |
| ReservationDate | U-P04 提交日期 |
| TimeSlot | 1/2/3 固定枚举 |
| StoredStatus | Reserved = 1 |
| CreatedAt | 本次 TimeProvider 本地时间 |
| CancelledAt | NULL |
| CancelledBy | NULL |
| RowVersion | SQL Server 自动生成 |

创建后影响：

1. 相同座位日期时段不可再产生 Reserved。
2. 当前用户相同日期时段不可预约其他座位。
3. 预约总数 +1。
4. 若预约日期为今天，今日非取消预约数 +1。

### 6.2 用户取消前后

| 字段 | 取消前 | 取消后 |
| --- | --- | --- |
| StoredStatus | Reserved | Cancelled |
| CancelledAt | NULL | now |
| CancelledBy | NULL | Student |
| RowVersion | 旧值 | 数据库新值 |

取消后影响：

1. 过滤唯一索引不再包含该记录，座位时段和用户时段立即释放。
2. 预约总数不变。
3. 若预约日期为今天，今日非取消预约数 -1。
4. 历史记录保留，U-P05 展示“已取消”。

## 7. 链路一：状态流转

| 起点 | 触发 | 存储状态 | 展示状态 | 页面结果 |
| --- | --- | --- | --- | --- |
| 无预约 | U-P04 合法提交 | Reserved | 已预约 | Redirect U-P05 |
| Reserved | 时间早于结束时间 | Reserved | 已预约 | U-P05 可查看 |
| Reserved | 时间到达结束时间 | Reserved | 已完成 | U-P05 动态显示，不写库 |
| Reserved | 学生在开始前取消 | Cancelled | 已取消 | U-P05 成功提示 |
| Reserved | 学生在开始后取消 | 不变 | 已预约或已完成 | U-P05 失败提示 |
| Cancelled | 重复取消 | Cancelled | 已取消 | U-P05 失败提示 |

座位 IsEnabled 与预约状态分开：Seat 停用不会改变已有 Reservation，但会使新的 CreateAsync 在第 3 步失败。

## 8. 链路一：异常与边界情况

| 场景 | 检测位置 | 系统处理 | 页面表现 |
| --- | --- | --- | --- |
| Session 无当前用户或用户已停用 | CurrentExperienceUserService | 回退默认启用用户；无可用用户则失败 | U-P01 提示体验账号不可用 |
| 座位不存在 | SeatService / ReservationService | 不写库 | U-P03 404 或 U-P02 提示 |
| 座位已停用 | ReservationService | 不写库 | U-P04 显示“座位已停用” |
| 日期早于今天或超过 7 天 | ReservationService | 不写库 | U-P04 保留输入并提示 |
| 当天时段已开始 | ReservationService | 不写库 | 对应时段不可选；手工 POST 仍被拒绝 |
| 座位冲突 | Service 预查 / Seat 过滤唯一索引 | 不新增 | U-P04 提示座位已被预约 |
| 用户冲突 | Service 预查 / User 过滤唯一索引 | 不新增 | U-P04 提示当前时段已有预约 |
| 用户双击提交 | Service / 唯一索引 | 最多 1 条 Reserved | 第二次请求提示冲突 |
| 用户取消他人预约 | 归属查询 | 返回统一不存在/无权错误 | U-P05 不泄露他人数据 |
| 用户取消已开始预约 | 开始时间校验 | 不更新 | U-P05 提示已不能取消 |
| 重复取消 | StoredStatus 校验 | 不更新 | U-P05 提示状态已变化 |
| 并发取消 | RowVersion | 后保存请求失败 | U-P05 提示刷新重试 |
| 数据库暂时失败 | 全局异常处理 | 记录日志，不显示堆栈 | 通用错误提示 |

## 9. 链路二：管理预约与状态回流

### 9.1 页面与路由

| 步骤 | 页面 | HTTP | 路由 | 成功去向 |
| --- | --- | --- | --- | --- |
| 1 | A-P01 管理员登录 | GET | `/admin/login` | A-P01 |
| 2 | A-P01 提交登录 | POST | `/admin/login` | Redirect `/admin/stats` |
| 3 | A-P04 统计页 | GET | `/admin/stats` | A-P04 |
| 4 | A-P02 预约管理 | GET | `/admin/reservations?date={date}&status={status}` | A-P02 |
| 5 | A-P02 管理员取消 | POST | `/admin/reservations/{id}/cancel` | Redirect 回当前 A-P02 筛选 |
| 6 | A-P04 查看变化 | GET | `/admin/stats` | A-P04 |
| 7 | 管理员退出 | POST | `/admin/logout` | Redirect `/admin/login` |

所有 Admin Controller 使用 `[Authorize(AuthenticationSchemes = "AdminCookie")]`；AuthController 的 Login 允许匿名。

### 9.2 Controller 动作

| Controller.Action | 处理 |
| --- | --- |
| `AuthController.Login(returnUrl)` GET | 返回登录页；仅接受本地 returnUrl |
| `AuthController.Login(AdminLoginViewModel)` POST | 校验账号，签发 AdminCookie，跳转 A-P04 |
| `AuthController.Logout()` POST | `SignOutAsync("AdminCookie")`，跳转 A-P01 |
| `StatsController.Index()` | 调用 StatisticsService 查询 5 个数字 |
| `Admin.ReservationsController.Index(date, status)` | 规范化页面提供的日期、状态条件，组装查询对象并查询列表 |
| `Admin.ReservationsController.Cancel(id, filter)` POST | 从 Claims 取得 AdminId，调用管理员取消并恢复筛选条件 |

取消后重定向使用明确的 date、status 路由值，不接受任意外部 returnUrl。area、experienceUserId 只保留在 Service/DataAccess 查询对象和测试中，首版 A-P02 不渲染对应控件。

### 9.3 Service 职责

```text
AdminAuthService.ValidateCredentialsAsync(username, password)
StatisticsService.GetOverviewAsync(today)
ReservationService.GetAdminListAsync(filter, now)
ReservationService.CancelByAdminAsync(reservationId, adminId)
```

`CancelByAdminAsync` 校验顺序：

| 顺序 | 校验/动作 | 失败错误 |
| --- | --- | --- |
| 1 | 管理员 Claims 中存在有效 AdminId | 认证中间件拦截或 `AdminUnavailable` |
| 2 | Reservation 存在 | `ReservationNotFound` |
| 3 | StoredStatus = Reserved | `ReservationAlreadyCancelled` |
| 4 | now 早于 ReservationDate + TimeSlot.EndTime | `ReservationCompleted` |
| 5 | 更新 StoredStatus、CancelledAt、CancelledBy = Admin | 无 |
| 6 | SaveChanges，RowVersion 未变化 | `ReservationChanged` |

AdminId 用于确认当前管理员身份和日志上下文；按数据库设计，本期不保存 `CancelledByAdminId` 外键，只保存 `CancelledBy = Admin`。

### 9.4 DataAccess 动作与动态状态筛选

A-P02 查询从 `Reservations.AsNoTracking()` 开始，投影 ExperienceUser 和 Seat 显示字段，再按非空条件组合过滤。

展示状态筛选必须转换为数据库条件，不先 `ToListAsync()` 再内存筛选：

```text
Cancelled:
  StoredStatus == Cancelled

Completed:
  StoredStatus == Reserved
  AND (
    ReservationDate < today
    OR (
      ReservationDate == today
      AND (
        TimeSlot == Morning   AND nowTime >= 12:00
        OR TimeSlot == Afternoon AND nowTime >= 18:00
        OR TimeSlot == Evening   AND nowTime >= 21:30
      )
    )
  )

Reserved:
  StoredStatus == Reserved
  AND NOT CompletedCondition
```

`today`、`nowTime` 在进入查询前由同一次 `TimeProvider.GetLocalNow()` 计算为参数，查询表达式只比较数据库列和参数，保证 EF Core 可翻译。

其他筛选：

```text
date != null             -> ReservationDate == date
area not empty           -> Seat.Area == normalizedArea
experienceUserId != null -> ExperienceUserId == experienceUserId
```

area、experienceUserId 是 Service/DataAccess 的查询能力和测试输入，不是首版 A-P02 页面控件。Controller 首版从页面接收 date、status，并为其余条件传空值。

默认排序：ReservationDate 降序、TimeSlot 降序、CreatedAt 降序。

### 9.5 数据库变化与状态回流

管理员取消成功：

```text
StoredStatus: Reserved -> Cancelled
CancelledAt: NULL -> now
CancelledBy: NULL -> Admin
RowVersion: old -> new
```

回流效果：

1. A-P02 原记录展示状态变为“已取消”，取消按钮消失。
2. 对应座位时段和用户时段从过滤唯一索引释放。
3. 预约总数不变。
4. 若 ReservationDate 为 today，今日非取消预约数减少 1。
5. 管理员返回 A-P04 时 StatisticsService 重新查询，不使用页面缓存数字。

### 9.6 管理链路异常与权限边界

| 场景 | 处理 |
| --- | --- |
| 未登录访问 A-P02/A-P04 | Cookie 中间件跳转 `/admin/login` |
| 登录密码错误 | 留在 A-P01，统一提示“账号或密码错误” |
| returnUrl 为外部地址 | 忽略并跳转 `/admin/stats` |
| 筛选用户或区域不存在 | 返回空列表，不抛异常 |
| 管理员取消已取消记录 | 不更新，提示状态已变化 |
| 管理员取消已结束记录 | 不更新，展示状态为已完成 |
| 取消时发生 RowVersion 冲突 | 不覆盖新状态，提示刷新后重试 |
| 退出后再次访问管理页 | Cookie 已失效，重新跳转 A-P01 |

## 11. 默认补足链路 / 当前假设

### 11.1 默认链路使用情况

未使用默认链路补足。前序文档已经明确用户预约主链路、用户取消闭环、管理员登录、管理预约和统计回流，本设计仅将其落实为工程步骤。

### 11.2 本轮细化假设

1. 系统设计中的工程路由优先于页面树早期的建议路径；页面编号和职责不变。
2. CurrentExperienceUserService 在 Web 边界读取 Session，Controller 取得用户编号后显式传给业务 Service。
3. DataAccess 不新增 Repository，Service 直接使用 AppDbContext 编写 LINQ。
4. 管理员取消不保存 AdminId 外键，只保存 CancelledBy = Admin，符合当前单管理员范围。
5. 唯一索引异常只在识别到约定索引名时转换为业务冲突，其他数据库异常不吞掉。
6. 业务时间统一取服务器本地时间，一次操作只读取一次 TimeProvider。
7. 管理端页面的日期、状态筛选条件通过路由值恢复，不使用可跳往外部地址的 returnUrl；区域、用户条件只保留在 Service/DataAccess 查询对象。
8. DbSeeder 只在核心表为空时写入整套演示数据；如需复位，删除并重建 LocalDB，不追加半套种子。
