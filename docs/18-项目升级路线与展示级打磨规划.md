# 图书馆座位预约系统项目升级路线与展示级打磨规划

> 规划日期：2026-07-14
>
> 文档性质：后续升级路线，不代表当前已经实现
>
> 主线目标：面试展示优先，同时让成果更接近真实可运行的商业产品原型

## 0. 规划边界

本文件只做升级规划，不修改代码、数据库、README 或既有交付文件。

必须先纠正一个前提：当前仓库“已经完成并可提交”只适用于课程文档与静态原型成果，不适用于可运行系统。README 和 docs/17 已明确：

- 没有 `.sln`、`.csproj` 和 `src/`；
- 没有 EF Core、Migration、LocalDB 和种子数据；
- 用户端与管理端业务完成度为 0%；
- 没有可用演示账号；
- 没有系统级自动化、兼容性和回归测试。

因此，第一轮升级不能从“增加高级功能”开始，必须先把现有设计压缩成一条真实可运行、可复现、可自动验收的纵向业务闭环。

## 1. 当前项目真实基线判断

### 1.1 当前已完成

1. 项目范围已固定：用户端 5 页、管理端 4 页。
2. 立项、MVP、PRD Lite、页面树、页面卡、系统设计、数据库设计和关键链路设计已形成文档。
3. static-v2 已覆盖 9 个核心页面，并包含空状态、冲突提示和后台错误等视觉状态。
4. 用户端手机优先、管理端桌面优先的 UI 方向较明确。
5. 数据设计已明确 4 个核心实体、3 个固定时段、状态、RowVersion 和两类预约冲突约束。
6. 原型截图、QA 结果、任务板、审计和复盘证据较完整。

### 1.2 当前尚未完成

1. .NET 8 MVC 工程和分层代码。
2. EF Core Code First、Migration、LocalDB 和幂等种子数据。
3. 体验用户 Session 和真实查座。
4. 预约创建、冲突、回看和取消。
5. 管理员登录、AdminCookie、预约处理、座位管理和统计。
6. 用户端与管理端基于同一数据库的状态回流。
7. build、run、Playwright、smoke、兼容性、回归和 Bug 闭环。
8. 独立仓库入口、最终 release 和可复现部署。

### 1.3 现在最像课堂项目的地方

1. 文档和原型很多，但没有可运行纵向切片。
2. 页面中的登录、预约、取消和统计都由静态假数据模拟。
3. 数据库约束、分层和状态规则停留在设计文档中，没有代码证据。
4. 任务板完整，但开发卡片完成率为 0%。
5. 展示依赖讲解文档，不能通过一次真实操作证明系统能力。

### 1.4 当前最有潜力继续拔高的地方

1. 业务范围小而完整，适合快速做出真实闭环。
2. 用户预约和管理员处理天然构成跨角色状态回流，面试展示效果强。
3. 两类冲突、动态 Completed、取消边界、RowVersion 和过滤唯一索引具备工程讨论价值。
4. static-v2 已有较完整视觉基础，可直接迁移为 Razor，而不需要重新设计。
5. 现有文档能直接作为 AI 生成 Entity、Service、ViewModel、测试和验收脚本的结构化输入。

## 2. 当前最影响展示级成熟度的 Top 问题

| 排名 | 问题 | 严重性 | 为什么最影响展示 |

## 3. 最值得优先投入的升级项

### 升级项 1：建立可运行的 .NET 8 纵向基础

**优先级：S0，第一件事。**

#### 当前问题表现

仓库没有 Solution、Web 项目、`Program.cs`、分层目录和构建结果，设计技术栈没有落地。

#### 为什么会拉低项目成熟度

没有可执行程序时，架构、数据库和测试都无法被验证。面试官只能评价文档表达，不能评价软件工程能力。

#### 对面试展示的价值

一次 `dotnet build` 和 `dotnet run` 成功，比继续增加几十页设计说明更能证明执行能力。项目会从“方案”变成“软件”。

#### 对接近商业化的价值

形成清晰的 Controller / Service / DataAccess / ViewModel / Entity 边界，为后续业务、测试、日志和部署提供稳定载体。

#### AI 可参与的工作点

- 根据 docs/07 生成 Solution、MVC 项目、目录和依赖注册。
- 根据 docs/08 生成 Entity、DbContext、Fluent Configuration 和 Migration 初稿。
- 根据 docs/09 生成 Service/DataAccess 接口和路由骨架。
- 自动扫描命名空间、依赖注入和 nullable 警告。

AI 生成后必须实际 build，并逐项核对索引、状态枚举和依赖方向。

#### 建议开发步骤

1. 先修订并复核 `CONS-01`、`CONS-07`、`CONS-11`。
2. 创建 `LibrarySeatReservation.sln` 和 `src/LibrarySeatReservation.Web`。
3. 安装 EF Core SQL Server、Design/Tools 包。
4. 注册 MVC、DbContext、Session、AdminCookie 和 TimeProvider。
5. 建立 4 个 Entity、ViewModel、Service 和 DataAccess 骨架。
6. 完成首次 build 和首页启动。

#### 关键产物

- Solution 和 Web `.csproj`；
- `Program.cs`；
- 分层目录与接口；
- build 日志；
- 首页、用户入口、管理员入口。

#### 最低验收线

1. clean clone 后 `dotnet restore` 和 `dotnet build` 退出码为 0。
2. `dotnet run` 后首页返回 HTTP 200。
3. 三个入口路由可访问，未实现动作有安全占位而非异常堆栈。

### 升级项 2：完成 Code First、种子数据和用户预约闭环

**优先级：S0，第一轮核心。**

#### 对面试展示的价值

可现场演示“切换用户 -> 查座 -> 预约 -> 我的预约 -> 取消”，并展示数据库记录和冲突约束，含金量远高于普通 CRUD。

#### 对接近商业化的价值

真实持久化、数据隔离、冲突约束和幂等初始化是产品可靠性的基础。

#### AI 可参与的工作点

- 生成 InitialCreate Migration 和 DbSeeder 初稿。
- 根据 2/1/8/3 口径生成相对日期种子数据。
- 生成用户端 Controller、Service、LINQ 查询和 Razor 表单。
- 生成异常映射和表单验证测试。

#### 建议开发步骤

1. 实现 4 表 Mapping、CHECK、RowVersion 和两个过滤唯一索引。
2. 创建 Migration 并更新 LocalDB。
3. 实现幂等种子：2 用户、1 管理员、8 座位、3 预约。
4. 实现 CurrentExperienceUserService 和 Session 切换。
5. 完成 U-P01 至 U-P03 的真实查库。
6. 完成 U-P04 创建预约和两类冲突。
7. 完成 U-P05 数据隔离、动态状态和用户取消。

#### 关键产物

- Migration 和 ModelSnapshot；
- DbSeeder；
- User/Seat/Reservation Service；
- U-P01 至 U-P05 Razor 页面；
- 数据库前后对比记录。

#### 最低验收线

1. 可从空数据库一条命令建表并初始化数据。
2. 至少一次预约真实写库、一次“我的预约”真实回查。
3. SeatConflict、UserConflict、停用座位和非法取消均有可读提示。
4. 取消后相同时段可重新预约。

### 升级项 3：完成管理员权限与状态回流闭环

**优先级：S0，第一轮展示亮点。**

#### 当前问题表现

#### 对接近商业化的价值

补齐后台访问控制、运营处理、资源维护和状态反馈，系统开始具备真实服务闭环。

#### AI 可参与的工作点

- 生成 A-P02 查询、取消、A-P03 编辑启停和 A-P04 统计代码。

#### 建议开发步骤

1. 实现管理员哈希登录、AdminCookie、退出和未登录跳转。
2. 完成 A-P02 日期/状态筛选和管理员取消。

#### 关键产物

- Admin Area 和 `_AdminLayout`；
- Auth/Reservations/Seats/Stats Controller；
- AdminAuthService、StatisticsService；
- 权限和状态回流测试；
- 真实管理员演示账号。

#### 最低验收线

1. 未登录访问 A-P02/A-P03/A-P04 必须跳 A-P01。
2. 正确账号登录、错误账号提示、退出失效均通过。
3. 管理员取消写回数据库，用户端刷新后状态变化。
4. 座位停用阻止新预约但不删除历史记录。

### 升级项 4：把 static-v2 迁移成一致的产品级 Razor 体验

**优先级：S1，业务闭环后立即做。**

#### 当前问题表现

9 页视觉已存在，但与未来 MVC 页面断开；静态按钮、假数据和状态参数不能直接成为业务 UI。

#### 为什么会拉低项目成熟度

如果工程页面退回 Bootstrap 默认模板，已有原型优势会丢失；如果直接复制假逻辑，又会出现前后端状态不一致。

#### 对面试展示的价值

真实数据驱动的 9 页界面、统一组件和响应式截图会让项目第一眼不像脚手架作业。

#### 对接近商业化的价值

统一导航、状态、反馈、空状态和操作区能够降低使用成本，也方便后续迭代。

#### AI 可参与的工作点

- 将 static-v2 拆成 Razor Layout、Partial 和页面模板。
- 将静态假数据替换为强类型 ViewModel。
- 生成状态标签、反馈、空状态和确认弹窗组件。
- 使用 Playwright 截图做 390px/1366px 视觉对比。

#### 建议开发步骤

1. 提取颜色、间距、按钮和状态样式到 `site.css`。
2. 建立 `_UserLayout`、`_AdminLayout` 和共享反馈 Partial。
3. 逐页迁移 U-P01 至 U-P05，再迁移 A-P01 至 A-P04。
4. 删除页面中的业务模拟 JavaScript，只保留必要交互。
5. 补成功、失败、空数据、加载异常和不可操作状态。
6. 做 Edge/Chrome、手机/桌面截图回归。

#### 关键产物

- 9 个真实 Razor 页面；
- 2 个 Layout；
- 共享状态与反馈组件；
- 自定义 CSS；
- 响应式截图与对比记录。

#### 最低验收线

1. 9 页均使用真实 ViewModel，不读取原型假数据。
2. 用户端 390px、管理端 1366px 无关键溢出和遮挡。
3. 空状态、错误状态和成功反馈至少各有真实数据触发样例。
4. 用户端与管理端状态颜色和文案一致。

### 升级项 5：把业务规则、错误、安全和可观测性做成工程证据

**优先级：S1，体现“不是只会 CRUD”。**

#### 当前问题表现

冲突、时间、取消、并发、安全、日志和配置都停留在文档，没有代码和运行证据。

#### 为什么会拉低项目成熟度

商业系统的成熟感主要来自错误情况下仍可预测，而不是正常路径能点通。

#### 对面试展示的价值

可以具体讲解数据库约束与 Service 校验双保险、TimeProvider 可测试性、RowVersion 并发控制、统一异常映射和结构化日志。

#### 对接近商业化的价值

降低重复预约、越权操作、并发覆盖、敏感信息泄露和线上排障困难。

#### AI 可参与的工作点

- 根据规则矩阵生成 Service 校验和错误类型。
- 生成 Exception Handler、Problem/反馈映射和日志模板。
- 生成 CSRF、Cookie、安全 returnUrl 和配置检查清单。
- 扫描 N+1、缺少 AsNoTracking、未参数化查询和敏感日志。

#### 建议开发步骤

1. 将时段映射、DisplayStatus、CanUserCancel、CanAdminCancel 集中到 Service。
2. 为业务结果定义稳定错误码，不用异常文本做判断。
3. 启用 AntiForgery、Cookie 安全配置和管理员密码哈希。
4. 使用 User Secrets/环境变量保存连接字符串和演示密码。
5. 使用 `ILogger` 记录登录、预约、取消和数据库异常，不记录密码。
6. 核对查询索引、AsNoTracking、投影和分页需求。
7. 增加 health endpoint 或启动自检，验证数据库可连接。

#### 关键产物

- 业务规则矩阵与实现；
- 统一结果/错误模型；
- 全局异常处理；
- 安全配置清单；
- 结构化日志样例；
- 查询与索引说明。

#### 最低验收线

1. 两类冲突、4 类非法取消和并发变化都有稳定结果码与页面反馈。
2. 用户不能读取或取消他人预约。
3. 管理页均受 AdminCookie 保护。
4. 日志能定位一次预约失败，且不包含密码或敏感数据。
5. 配置不硬编码机器绝对路径或明文密钥。

### 升级项 6：建立自动化质量门禁

**优先级：S1，展示稳定性证据。**

### 升级项 7：实现一键复现、环境配置和可演示部署

7. 发布 release，并在 P0 清零后创建版本标签。

#### 关键产物

- 部署说明；

### 升级项 8：把工程亮点整理成面试证据包

**优先级：S2，在系统可运行后做。**

#### 当前问题表现

亮点散落在大量阶段文档中，面试官难在几分钟内理解项目难点、个人贡献和工程取舍。

作品本身即使完成，如果没有清晰证据和故事，也容易被看成普通课程作业。

#### 对接近商业化的价值

- 生成 3 分钟演示脚本、README 截图布局和面试问答。

- ADR；

#### 最低验收线

2. 所有截图、指标和功能均可在仓库中追溯。

## 5. 不建议现在投入的过度设计项

在第一轮闭环完成前，不建议投入：

1. 微服务、DDD 全套分层、CQRS、事件溯源。
2. Redis、消息队列、分布式锁、Kubernetes。
3. 多租户、多馆区联动和复杂组织权限。
4. Identity 全量账号体系、第三方登录和短信验证码。
5. 支付、扫码签到、消息推送和邮件通知。
6. 移动 App、小程序和前后端分离重写。
7. AI 座位推荐、热力预测和复杂 BI 大屏。
8. 数据仓库、搜索引擎、审计平台和全链路追踪系统。
9. 在没有真实流量前做缓存、分库分表和大规模压测。

这些能力不是永远不做，而是当前无法解决最核心的“系统还不能运行和闭环”问题。AI 可以加速生成代码，但不能替代优先级判断和真实验收。

## 6. AI 辅助推进原则

1. 每次只把 1 至 3 张相邻任务卡交给 AI，避免一次生成整个系统后无法定位问题。
2. 先让 AI 从 docs/08、docs/09 提取契约，再生成代码，防止命名漂移。
3. 每生成一层立即 build；每生成一个写操作立即查库验证。
4. Migration、索引、权限和取消规则必须由测试或数据库结果验证，不能只读代码。
5. UI 生成后用 Playwright 截图和像素/溢出检查，不靠主观描述。
6. AI 生成的 README、账号、命令和指标只能写入实测结果，禁止提前填充。
7. 每个可保留节点单独 commit，使失败时能快速定位和回退。

## 7. 最终结论

### 如果只继续打磨一轮，第一优先级应该做什么

**第一优先级是完成“真实预约 + 管理取消 + 用户回看”的可运行纵向切片，并让新环境可以一键建库和启动。**

具体不是只做一个首页，也不是先把 9 页全部美化，而是：

1. 清零开发前 P0；
2. 创建 .NET 8 MVC + EF Core 工程；
3. 建库并初始化最小数据；
4. 用户完成一次真实预约写入和回看；
5. 管理员登录并取消该预约；
6. 用户端看到状态和时段变化；
7. 用 Playwright msedge 自动跑通这条链路；
8. README 提供可复现命令和真实账号。

这一个闭环完成后，现有文档、数据库设计和 static-v2 才会同时转化为面试价值。

### 最能让面试官感受到项目成熟度的 3 个证据

1. **可复现证据**：从干净 clone 到 build、Migration、seed、run 一次成功，有明确命令和版本。
2. **跨角色真实数据证据**：用户预约、管理员取消、用户回看、时段释放和统计变化在同一数据库中连续演示。
3. **质量证据**：Service/数据库规则测试 + Playwright msedge 主链路 + CI/回归报告，能展示冲突、权限和异常不只在正常路径下工作。

当这 3 个证据都能从仓库直接验证时，项目才会从“文档完整的课堂原型”升级为“面试中有工程含金量的真实产品原型”。
