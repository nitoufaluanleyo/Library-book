# 图书馆座位预约系统《PRD Lite》审计

审计对象：`docs/03-PRD-Lite.md`

审计说明：本文件只记录审计结果，不直接修改原文件。

## A. 总体判断

这份《PRD Lite》整体通过，但建议完成少量口径修正后再正式冻结页面树。

用户端和管理端功能均已存在。用户端包含体验账号切换、首页、座位列表、座位详情与预约、冲突校验、我的预约与取消；管理端包含登录保护、预约管理、座位管理和基础统计。两端都给出了最小验收标准，不是只有功能名称。

文档形成了完整主流程：体验账号切换 → 查看座位 → 选择日期与时段 → 服务端校验 → 预约成功或失败 → 查看和取消预约；管理端也形成登录 → 查看/筛选预约 → 取消预约 → 维护座位 → 查看统计的闭环。

页面严格控制为 P01 至 P09 共 9 个主要页面或页面组，符合上一阶段“4 个用户端页面 + 5 个管理端页面”的 MVP 口径。预约状态使用“已预约、已取消、动态已完成”，座位状态使用“启用、停用”，与图书馆座位预约题目匹配。

文档没有混入支付、扫码、消息推送、真实账号、多馆区、复杂图表等超纲功能。新增的取消方、取消时间、密码哈希和服务端校验属于保证主流程可检查所需的数据与规则，不构成范围扩大。

## B. 发现的问题清单

| 编号 | 发现的问题 | 严重程度 | 审计说明 |
| --- | --- | --- | --- |
| 1 | P07 与 P08 都提供座位启停入口 | 中 | P07 页面清单写有“快速启停”，P08 共享表单也包含启停状态。两个入口会增加重复处理、提示和验收分支，与“列表 + 共享表单”的收束方向不完全一致。 |
| 2 | 停用座位是否能进入 P03 详情页不够明确 | 中 | U03 写“点击启用座位可以进入详情”，P02 页面清单对所有座位统一写“查看详情”，U04 又要求停用座位提交时由服务端拒绝。页面树需要明确停用座位是否有详情入口。 |
| 3 | “今日有效预约数”的名称与计算口径不完全匹配 | 中 | 指标按“预约日期为当天且未取消”计算，因此当天已经结束、展示为“已完成”的记录也会被计入。“有效”容易被理解为仅包含当前仍可使用的预约。 |
| 4 | 重复提交的验收表述不够严格 | 中 | U04 写“不应无提示地产生重复预约”，该表述理论上允许“提示后仍产生重复记录”。实际业务规则应明确为第二次请求被拒绝，始终只保留一条有效预约。 |
| 5 | AT10 的执行动作不够固定 | 低 | AT10 使用“等待或准备一条结束时间已过的有效预约”，验收人员可能不知道如何准备。文档已经规定一条动态已完成种子预约，应直接指定使用该记录验收。 |
| 6 | “我的预约”倒序规则不利于优先查看最近待使用记录 | 低 | 按预约日期倒序会把最远的未来预约放在前面。课堂数据少时不影响功能，但更自然的规则是未完成预约按日期升序、历史记录按日期倒序。 |

## C. 必改项

以下事项应在下一步“页面树与业务流程”开始时先固定。可以在下一步文档中明确，不要求直接回改本次审计对象。

1. 座位启停只保留一个主要操作入口。
   建议 P07 只展示状态并提供“编辑”，由 P08 统一修改启停状态。这样少一套 POST 动作，也更符合单人课堂项目的工作量。

2. 统一停用座位的详情访问规则。
   建议停用座位仍可从 P02 进入 P03 查看基础信息，但 P03 隐藏或禁用预约按钮并显示“座位已停用”；无论页面如何显示，服务端提交仍必须拒绝停用座位。

3. 修正统计指标名称或计算方式。
   建议保留现有简单计算方式，将页面名称统一为“今日非取消预约数”，明确包含当天已完成记录，避免增加实时占用统计逻辑。

4. 收紧重复提交验收口径。
   统一写为：同一请求再次提交时必须被座位冲突或用户冲突规则拒绝，数据库始终只保留一条有效预约。

5. 固定动态完成状态的验收数据。
   AT10 直接使用初始化器生成的“结束时间已过但 StoredStatus 仍为 Reserved”的种子记录，不要求课堂验收时等待或手工改数据库。

## D. 建议优化项

1. 在下一步页面树中为 P01 至 P09 分别映射对应的 U/A 功能编号和 AT 验收编号，形成可追踪关系。
2. “我的预约”建议把未开始或进行中的预约放在前面，历史已取消和已完成记录放在后面。
3. 为 P02、P04、P06、P07 明确无数据时的空状态文字，但不要新增独立空状态页面。
4. 页面流程中明确登录成功固定进入 P09，管理端公共导航提供 P06、P07、P09 和退出入口。
5. 后续技术设计应考虑重复点击提交的防护，例如提交后跳转或禁用按钮；无论采用何种做法，服务端冲突校验仍是最终保障。
6. README 中应提供重建演示数据库步骤，保证 3 条相对日期种子预约可以恢复到可验收状态。

## E. 是否可以进入下一步“页面树与业务流程”

可以有条件进入下一步“页面树与业务流程”。

进入下一步时，先按 C 部分固定座位启停入口、停用座位详情入口、统计指标名称、重复提交结果和动态完成验收数据，再绘制页面节点与流程连接。上述问题属于页面动作和验收措辞精度，不改变 9 页范围，不增加数据对象，也不需要重新进行 MVP 评审。

最终结论：用户端与管理端功能完整，主流程闭环，验收标准总体具体，状态流转合理，无明显超纲需求，与上一阶段 MVP 保持一致。完成上述小范围口径修正后，可以直接进入“页面树与业务流程”。
