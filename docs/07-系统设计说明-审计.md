# 《系统设计说明》审计报告

## A. 总体判断

**审计结论：有条件通过。**

文档已经形成可执行的 ASP.NET Core MVC 分层方案。Controller、Service、DataAccess、ViewModel、Entity 的职责和依赖方向基本清楚；用户端 5 页、管理端 4 页均有对应的 Razor View、GET/POST Action 和 ViewModel；预约、取消、座位维护、统计等主链路也已落实到具体 Service 方法。

方案总体适合单人课堂项目：采用单解决方案、单 Web 项目和 Admin Area，没有引入微服务、前后端分离、Identity、通用 Repository 或自定义 UnitOfWork。目录虽然较完整，但仍在可控范围内。

当前没有阻塞数据库设计的结构性错误。不过，身份上下文、DataAccess 定义、时间来源和取消方字段仍有少量职责或口径歧义，必须在数据库设计和代码骨架开始前确定。

## B. 发现的问题清单

| 编号 | 严重程度 | 发现的问题 | 影响 |
| --- | --- | --- | --- |
| B-01 | 中 | Session 身份读取与业务 Service 的边界不够统一 | 第 4.3 节把 `CurrentExperienceUserService` 放在 Service 层并负责读取 Session，第 7.1 节又写 `ReservationService.CreateAsync` 从 Session 对应服务获取用户；但取消方法又显式接收 `currentUserId`。若实现时让 ReservationService 依赖 Session，将降低可测试性并让业务层依赖 HTTP。 |
| B-02 | 中 | DataAccess 的名称与实际职责表述存在轻微冲突 | 文档目标写“数据库访问集中在 DataAccess”，同时又明确 Service 直接通过 `AppDbContext` 编写查询。当前 DataAccess 实际是 EF Core 持久化基础设施层，不是包含查询实现的独立数据访问层，需要统一说法。 |
| B-03 | 中 | 取消方的数据落点尚未完全明确 | 文档同时出现 `CancellationActor`、管理员 `adminId` 和“取消方通过业务字段记录”。下一步必须确定只保存取消方类型，还是同时保存管理员/体验用户编号，否则 Entity 与 Service 方法签名可能不一致。 |
| B-04 | 中 | 预约冲突的数据库最终防线尚未具体化 | 文档要求唯一约束兜底，但“已取消记录不占用时段”意味着普通唯一索引不能直接覆盖全部预约。下一步需确定过滤唯一索引或事务内校验方案。该问题属于数据库设计任务，但不能遗漏。 |
| B-05 | 低 | 时间来源存在两种表达 | 目录建议包含自定义 `SystemClock.cs`，主链路方法又直接传入 `now`。如果两种方式同时存在，可能出现不同页面使用不同当前时间的情况。 |
| B-06 | 低 | 每个 Service 都建立接口可能略重 | 5 组接口和实现对课堂项目仍可接受，但若接口只被一个实现使用且没有测试替身需求，会增加文件数量。 |
| B-07 | 低 | `ServiceResult` 的返回类型没有进一步约定 | 如果只设计一个非泛型结果类，查询数据、字段错误和业务错误可能逐渐塞入同一对象。建议至少区分 `ServiceResult` 与 `ServiceResult<T>`。 |
| B-08 | 低 | A-P03 编辑面板的打开方式只描述到 `editId` | GET 能加载编辑数据，但文档未明确 Razor 渲染后如何自动打开 Offcanvas。该问题不影响路由成立，开发时需要一个很小的页面脚本或条件样式处理。 |

### 已通过检查

1. 请求依赖方向明确为 Controller → Service → AppDbContext → LocalDB。
2. Controller 明确不直接访问 DbContext、不编写冲突规则、不向 View 传 Entity。
3. Service 统一负责预约、冲突、取消和展示状态，业务规则没有散落到 Razor。
4. DataAccess 包含 DbContext、Fluent API、Migration 和种子数据，边界基本合理。
5. ViewModel 按页面组织，Entity 不包含 Session、Cookie、TempData 和页面样式。
6. Admin Area 能清楚隔离管理端 Controller、View、路由和 Cookie 授权。
7. 9 个固定页面全部有工程映射，没有多出成功页、确认页、管理首页或独立编辑页。
8. 用户预约、用户取消、管理员取消、座位维护和统计均有完整工程落点。
9. Session 与管理员 Cookie 已区分，POST、防伪令牌、PRG 和异常处理原则明确。
10. 明确不采用通用 Repository、微服务、Identity 和前后端分离，避免了主要过度设计风险。
11. 统计范围已收束为 PRD 规定的 5 个数字，没有把原型趋势图扩展为后端需求。

## C. 必改项

以下口径必须在数据库设计完成前或代码骨架开始前确定：

1. **统一当前体验用户的传递方式。** 建议由 Controller 或独立的 Web 上下文访问器读取 Session 并得到 `currentUserId`，随后显式传给 `ReservationService`、`SeatService` 等业务方法。业务 Service 不直接依赖 `HttpContext` 或 Session。
2. **统一 DataAccess 的定义。** 本项目建议保持简单：DataAccess 只承载 DbContext、实体配置、Migration 和种子数据，具体 LINQ 查询写在 Service。文档和后续代码不得同时声称存在独立 Repository 查询层。
3. **确定取消审计字段。** 数据库设计必须明确 `CancellationActor`、`CancelledAt` 以及是否需要 `CancelledByAdminId`/`CancelledByExperienceUserId`，并与两个取消 Service 方法保持一致。
4. **确定冲突约束方案。** 数据库设计需要分别处理“同座位同日期同时段”和“同用户同日期同时段”两类冲突，并保证已取消记录可以重新预约。
5. **统一时间来源。** 建议使用 .NET 8 的 `TimeProvider` 注入 Service，种子数据、已完成状态、过期时段和取消判断都使用同一时间来源，不再额外维护含义重复的 `SystemClock`。

## D. 建议优化项

1. 为写操作使用 `ServiceResult`，为需要返回页面数据的操作使用 `ServiceResult<T>`，并固定少量错误代码。
2. 只为需要替换或单元测试的 Service 保留接口；不要求为了目录对称给每个简单类都创建接口。
3. 在数据库设计文档中增加“Entity 字段 - ViewModel 字段 - 页面位置”对照表。
4. 明确 A-P03 在存在 `editId` 时由条件脚本自动打开 Offcanvas，保存失败时保留输入和校验提示。
5. 将 ReservationService 的冲突、展示状态和两种取消规则作为首批单元测试，不扩大到全页面测试。
6. `ServiceCollectionExtensions` 可等注册代码明显变长后再提取，初期直接写在 `Program.cs` 更便于基础一般的学生理解。
7. 管理端授权可使用一个 Admin 基类 Controller 或 Area 级约定统一配置，避免每个 Controller 漏写认证方案。

## E. 是否可以进入下一步“数据库设计”

**可以有条件进入。**

分层、模块、页面映射和主链路均已达到数据库设计所需的清晰度，不需要重新设计整体架构。下一步数据库设计必须优先落实以下内容：

1. `ExperienceUser`、`Admin`、`Seat`、`Reservation` 四个实体的字段和关系。
2. 两类预约冲突的约束与事务处理方式。
3. 存储状态与动态展示状态的边界。
4. 取消时间、取消方类型和取消操作者编号的存储方式。
5. 统一时间来源及相对日期种子数据。

完成上述口径后，可以按当前系统设计直接进入 MVC 工程和数据库开发，不需要增加新的架构层或业务模块。
