# 图书馆座位预约系统《页面树与业务流程》审计

审计对象：`docs/04-页面树与业务流程.md`

审计说明：本文件只记录审计结果，不直接修改原文件。

## A. 总体判断

这份《页面树与业务流程》整体通过。

页面总数没有失控，严格保持用户端 5 页、管理端 4 页，共 9 页。页面状态、表单状态、提示和确认操作均明确为页面内行为，没有被重复计算为新页面。

用户端从首页、座位列表、座位详情、预约提交到我的预约完整存在；管理端从管理员登录、预约管理、座位管理到统计页完整存在。用户预约成功、预约失败、用户取消、账号切换、管理员登录、预约处理、座位维护和退出均有明确去向。

预约状态采用 Reserved、Cancelled 两种存储状态和“已预约、已取消、动态已完成”三种展示状态；座位采用启用、停用状态；管理员采用未登录、已登录状态。三组状态与图书馆座位预约业务匹配，流转条件和操作页面基本清楚。

文档没有结构性缺页或实质性重页。U-P03 与 U-P04 已明确拆分信息查看和预约表单职责，A-P03 使用同一页面的列表、新增、编辑状态承载座位维护，符合本期页面口径。

## B. 发现的问题清单

| 编号 | 发现的问题 | 严重程度 | 审计说明 |
| --- | --- | --- | --- |
| 1 | 当前体验用户无效时的恢复路径不够明确 | 中 | 预约失败链路会校验当前体验用户，但没有明确 Session 失效、Session 中用户已停用或系统没有启用体验用户时应跳到哪里。一直留在 U-P04 只显示错误，不利于恢复主流程。 |
| 2 | 直接访问 U-P04 时的异常跳转存在两个候选去向 | 中 | 用户端边界写“无效或停用座位时返回 U-P03 或 U-P02”，但没有逐项固定。座位不存在时无法正常显示 U-P03，停用座位则可以显示 U-P03，二者应分开。 |
| 3 | A-P03 的同页新增/编辑存在被开发成多页的实施风险 | 中 | 文档已要求 A-P03 承载列表、新增、编辑三个状态，但建议路径只有 `/Admin/Seats`，尚未说明页面卡应如何表现状态。若后续直接使用 MVC 脚手架，容易重新拆成 Create、Edit 两个页面。 |
| 4 | 页面树图中退出分支展示不完全一致 | 低 | 管理端公共导航和跳转总表都说明“任一管理页可以退出”，但树形图只在 A-P04 下画出退出到 A-P01，A-P02、A-P03 下没有画出该分支。规则本身存在，只是树图表达略有缺口。 |
| 5 | 部分返回动作依赖公共导航而非树形分支 | 低 | 例如 U-P05 可通过公共导航进入 U-P02，正文链路也使用了该路径，但用户端树形图只画出 U-P05 到 U-P03。不会造成实际断链，但页面卡需要统一体现公共导航。 |

## C. 必改项

以下事项应在下一步“页面卡与 UI 规范”开始时固定，可以写入对应页面卡，不要求直接回改本次审计对象。

1. 固定体验用户失效的恢复规则。
   建议 Session 中没有有效体验用户时，系统自动选择第一个启用体验用户并返回 U-P01；如果系统没有任何启用体验用户，则 U-P01 显示空状态，并禁止进入预约提交链路。

2. 固定 U-P04 异常访问的跳转规则。
   建议座位编号不存在时返回 U-P02 并提示“座位不存在”；座位存在但已停用时返回 U-P03 并提示“座位已停用”；只有有效且启用的座位可以正常显示 U-P04。

3. 固定 A-P03 为一个页面而不是三个页面。
   页面卡必须明确列表、新增、编辑是 A-P03 的三种同页状态，可使用同页表单区、局部视图或对话框，但不得新增独立座位新增页和编辑页。

4. 在管理端公共布局中统一退出入口。
   A-P02、A-P03、A-P04 都应显示退出入口并跳转 A-P01，不需要为退出增加新页面。

## D. 建议优化项

1. 下一步为所有跳转增加简短编号，例如 T01、T02，便于页面卡引用进入和离开条件。
2. 每张页面卡都应列出公共导航，避免只依据树形图时漏掉首页、座位列表、我的预约或退出入口。
3. U-P04 提交成功后继续采用跳转到 U-P05 的方式，刷新 U-P05 不重复发送预约请求。
4. U-P05 建议明确排序：待使用记录按时间升序，已取消和已完成记录按时间倒序。
5. A-P03 保存失败时应保持新增或编辑状态及用户已填内容，保存成功后才返回列表状态。
6. 为 U-P02、U-P05、A-P02、A-P03 的空数据状态准备页内提示，不新增空状态页面。

## E. 是否可以进入下一步“页面卡与 UI 规范”

可以进入下一步“页面卡与 UI 规范”。

进入下一步时，先把 C 部分的 4 项规则写入对应页面卡，再定义页面区域、字段、按钮、反馈和响应式布局。上述问题不会改变 9 页总量，不需要增加页面，也不影响用户端和管理端的主业务闭环。

最终结论：页面规模可控，两端页面完整，主链路清楚，状态流转合理，没有结构性缺页、重页或职责重复。补齐异常恢复和同页状态约束后，可以直接进入“页面卡与 UI 规范”。
