## 3. 范围一致性检查

### 3.1 统一范围

1. 项目题目统一为《图书馆座位预约系统》。
2. 技术栈统一为 ASP.NET Core MVC (.NET 8) + Razor + SQL Server LocalDB + EF Core 8 + Bootstrap。
3. 用户端固定 5 页，管理端固定 4 页，共 9 页。
4. 用户端功能固定为体验账号切换、查座、详情、预约提交、我的预约和页内取消。
5. 管理端功能固定为轻量登录、预约管理、座位管理和 5 项基础统计。
6. 核心数据固定为 ExperienceUser、Admin、Seat、Reservation 四个实体。
7. 本期继续不做支付、扫码、消息推送、真实学生账号、复杂权限、多馆区、审批、改签、批量处理、复杂图表和报表导出。

### 3.2 范围修订结果

`docs/01`、`docs/02`、`docs/03` 中的旧页面分组已同步为 5+4。取消预约仍是 U-P05 页内动作，座位新增/编辑仍是 A-P03 同页状态，没有增加页面。种子数据从早期区间值收口为后续已冻结的 2 个体验用户、1 个管理员、8 个座位和 3 条预约，没有扩大数据模型。

## 5. 数据一致性检查

### 5.1 实体与关系

| 实体 | 关键字段 | 关系或用途 |
| --- | --- | --- |
| ExperienceUser | Id、DisplayName、IsEnabled | 一对多 Reservation；当前 Id 存 Session |
| Admin | Id、Username、PasswordHash、IsEnabled | 预置管理员认证，不建复杂权限关系 |
| Seat | Id、SeatNumber、Area、Remark、IsEnabled、时间戳 | 一对多 Reservation；SeatNumber 唯一 |
| Reservation | ExperienceUserId、SeatId、ReservationDate、TimeSlot、StoredStatus、取消字段、RowVersion | 关联用户和座位，保留取消和完成历史 |

### 5.2 状态、查询与约束

1. StoredStatus 只保存 Reserved、Cancelled；Completed 只动态展示。
2. 学生开始前可取消，管理员结束前可取消；Seat 停用不自动取消已有预约。
3. A-P02 页面只暴露 date、status；Service/DataAccess 查询对象保留 area、experienceUserId。
4. 第五项统计统一命名为“今日非取消预约数”，口径为 `ReservationDate == today && StoredStatus == Reserved`，包含当天动态完成记录。
5. 两个过滤唯一索引统一使用正式字段：`SeatId + ReservationDate + TimeSlot`、`ExperienceUserId + ReservationDate + TimeSlot`，过滤 `StoredStatus = Reserved`。
6. 动态时间规则统一注入 .NET 8 TimeProvider，不自建第二套 SystemClock。

## 6. 实现一致性检查

请求依赖保持为：Browser -> Controller -> Service -> AppDbContext/DataAccess -> LocalDB。Controller 不直接访问 DbContext，Razor 不直接绑定 Entity，DataAccess 不增加通用 Repository。

U-P04 GET 已统一为：Controller 从 CurrentExperienceUserService 取得 currentExperienceUserId，再调用 `ReservationService.BuildCreatePageAsync(currentExperienceUserId, seatId, date, slot)`；Service 同时查询座位占用和当前用户时段占用。U-P03 只通过 SeatService 返回座位详情。

创建预约、用户取消和管理员取消均能对应 Reservation 字段变化、过滤唯一索引和 RowVersion。A-P03 只调用 SeatsController + SeatService 保存同页表单；A-P04 只调用 StatisticsService 查询 5 个实时数字。
