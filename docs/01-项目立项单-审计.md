# 图书馆座位预约系统项目立项单审计

审计对象：`docs/01-项目立项单.md`

## A. 总体判断

这份《项目立项单》整体可以通过。

项目目标清楚，技术栈、用户端、管理端、暂不开发内容都写得比较明确，适合作为课堂第三阶段“小项目开工”的基础文件。文档没有把范围扩大到支付、扫码、复杂账号、多馆区等高风险内容，符合单人实习生项目的控制要求。

页面规模控制在 10 个左右，基本适合单人课堂项目。成功标准大部分都能通过页面操作演示出来，风险也贴近课堂现实。

目前主要问题不是方向错误，而是有几处规则还偏粗，需要在下一步《需求分析与 MVP 确认》中补细，否则开发时容易出现理解不一致。

## B. 发现的问题清单

| 编号 | 问题 | 严重程度 | 说明 |
| --- | --- | --- | --- |
| 1 | 预约时间段规则还不够明确 | 中 | 文档提到“选择预约日期和时间段”，但没有说明时间段是固定选项、半小时一段、上午/下午/晚上，还是用户自由选择。 |
| 2 | 预约状态没有提前定义 | 中 | 成功标准和管理端都提到预约记录、取消预约、状态变化，但没有明确状态有哪些，例如待使用、已取消、已完成、已过期。 |
| 3 | 管理端“预约管理”的操作边界略模糊 | 中 | 文档写了“按状态进行基本管理”，但没有说明管理员能做哪些动作，是查看、取消、确认，还是只筛选。 |
| 4 | 座位管理可能导致页面数量膨胀 | 中 | 如果按 MVC Scaffold 默认做新增、编辑、详情、删除多个页面，页面数量会超过“10 个左右”的控制目标。 |
| 5 | 成功标准缺少“重复预约拦截”的演示动作 | 中 | 文档在关键限制流程中写了同一座位同一时间段不能重复预约，但成功标准里没有要求演示这个检查。 |
| 6 | 管理员登录成功标准还可以更可检查 | 低 | 目前写“输入管理员账号后可以进入后台”，建议补成“错误账号不能进入，正确账号可以进入”。 |
| 7 | 数据初始化要求还可以更具体 | 低 | 文档提到保留种子数据，但没有说明至少需要几个座位、几个体验用户、一个管理员账号。 |

## C. 必改项

这些内容建议在下一步《需求分析与 MVP 确认》中必须明确，不一定要回改当前立项单。

1. 明确预约时间段规则。
   建议采用固定时间段，避免复杂日历控件。例如：上午、下午、晚上三个时间段，或 08:00-12:00、14:00-18:00、18:30-21:30。

2. 明确预约状态。
   建议至少包含：已预约、已取消、已完成。若时间允许，再增加已过期。

3. 明确重复预约判断。
   建议规则为：同一座位、同一日期、同一时间段，只允许存在一条有效预约。

4. 明确管理端预约管理动作。
   建议本期只做查看、筛选、取消预约，不做复杂审批流程。

5. 控制座位管理页面数量。
   建议座位管理采用“列表页 + 新增/编辑表单”的简单做法，不额外扩展大量详情页和删除确认页。

## D. 建议优化项

1. 在需求分析阶段补一份“页面字段表”。
   例如座位列表显示：座位编号、区域、状态、容量、操作按钮。

2. 在需求分析阶段补一份“演示脚本”。
   按照用户预约、取消预约、管理员查看、停用座位、统计变化的顺序写，后面验收会更轻松。

3. 种子数据建议提前固定。
   建议至少准备 2 个体验用户、1 个管理员、8 到 12 个座位、2 到 3 条预约记录。

4. 统计页保持简单。
   建议只做数字卡片，不做复杂图表。统计项控制为座位总数、可用座位数、停用座位数、预约总数、今日预约数。

5. README 要尽早准备。
   LocalDB 和 EF Core 迁移对课堂同学来说容易卡住，建议后续项目骨架完成时就写运行命令。

## E. 是否可以进入下一步“需求分析与 MVP 确认”

可以进入下一步。

进入下一步时建议重点确认以下内容：

1. 预约时间段如何设计。
2. 预约状态有哪些。
3. 每个页面有哪些字段和按钮。
4. 用户端和管理端各自的最小演示流程。
5. 数据库最少需要哪些表和种子数据。

结论：本立项单方向正确、范围可控，可以作为下一阶段《需求分析与 MVP 确认》的输入文件。
