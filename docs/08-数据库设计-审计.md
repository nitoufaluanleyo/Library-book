# 《数据库设计》审计报告

## A. 总体判断

**审计结论：有条件通过。**

数据库设计与 `docs/03-PRD-Lite.md`、`docs/04-页面树与业务流程.md`、`docs/07-系统设计说明.md` 的主体口径一致。文档保留了前序已确认的 `ExperienceUser`、`Admin`、`Seat`、`Reservation` 四个核心实体，没有擅自套用默认 6 实体，也明确说明区域、固定时段和预约日志不拆表。

表关系、状态字段、管理筛选、五项统计和两类预约冲突都有明确落点。SQL Server 过滤唯一索引能够同时满足“有效预约不可冲突”和“取消后可以重新预约”，适合 LocalDB + EF Core 的单人课堂项目。

当前没有需要推翻数据模型的问题。进入代码前需要修正一处字段简称，并在关键链路详细设计中固定 EF Core 过滤索引写法、动态状态查询表达式和种子初始化幂等规则。

## B. 发现的问题清单

| 编号 | 严重程度 | 发现的问题 | 审计判断 |
| --- | --- | --- | --- |
| B-01 | 中 | 第 9.1 节约束汇总写成 `SeatId + Date + TimeSlot`、`UserId + Date + TimeSlot` | 与正式字段 `ReservationDate`、`ExperienceUserId` 不完全一致，容易在 Migration 或代码中产生错误命名。前文索引定义使用的是正确字段。 |
| B-02 | 中 | 过滤唯一索引是 SQL Server 专用实现，但尚未给出 EF Core 配置口径 | 需要在后续明确枚举转 `tinyint`、固定数值和 `HasFilter("[StoredStatus] = 1")`，否则 Migration 可能只生成普通唯一索引。 |
| B-03 | 中 | 管理端按动态展示状态筛选的业务口径清楚，但 LINQ 落法尚未固定 | “已完成”依赖日期、时段结束时间和当前时间。下一步需要给出可被 EF Core 翻译的条件，避免先加载全部数据再在内存筛选。 |
| B-04 | 中 | 相对日期种子数据未明确幂等判定 | 若应用每次启动都插入 3 条预约，会不断增加重复演示数据。初始化器需要通过标记、记录存在判断或仅空库初始化保证只执行一次。 |
| B-05 | 低 | `CancelledBy` 只保存 Student/Admin，不保存操作者编号 | 文档已经在“默认补足项 / 当前假设”中显式说明，且本期只有 1 个管理员、学生只能取消自己的预约，因此当前范围合理。扩展多管理员时才需要调整。 |
| B-06 | 低 | 时间字段统一保存服务器本地时间 | 对单机 LocalDB 课堂项目可接受，也已显式声明；实现时必须全部使用同一 `TimeProvider`，不能部分使用 `DateTime.Now`、部分使用 UTC。 |
| B-07 | 低 | RowVersion 只加在 Reservation，没有加在 Seat | 当前只有 1 个管理员，座位并发编辑风险很低；Reservation 的重复取消风险更高，当前取舍合理。 |
| B-08 | 低 | 管理端组合筛选没有为所有组合建立复合索引 | 当前只有 8 个座位和少量预约，已有 Date/Status、User/Date、Seat/Date 索引足够，不应为课堂数据量继续堆索引。 |

### 已通过检查

1. 核心实体严格沿用前序 4 实体，没有改成固定 6 实体。
2. `DisplayName`、`SeatNumber`、`Area`、`Remark`、`ReservationDate`、`TimeSlot`、`StoredStatus`、`CancelledAt`、`CancelledBy` 等字段命名与 PRD 一致。
3. ExperienceUser 与 Reservation、Seat 与 Reservation 的一对多关系清楚，外键删除行为使用 Restrict。
4. Admin 不建立预约外键属于显式设计决定，符合当前轻量管理员范围。
5. 座位列表、座位详情、预约提交、我的预约、预约管理、座位管理和统计页均有数据支撑。
6. 预约只存 Reserved、Cancelled；Completed 动态计算，未错误增加第三个存储状态。
7. 管理端可以按日期、展示状态、区域和体验用户组合筛选。
8. 五项统计均有明确数据来源和计算口径，取消预约不会减少预约总数。
9. 两个过滤唯一索引分别覆盖座位冲突和用户冲突，且只约束 Reserved 记录。
10. 取消字段 CHECK、枚举合法性 CHECK、RowVersion 和双层校验原则完整。
11. 默认补足项与当前假设已单列，RowVersion、TimeProvider、8 个座位和相对日期种子数据的理由合理。
12. 未引入触发器、存储过程、统计快照、复杂区域表或日志表，工作量适合单人课堂项目。

## C. 必改项

以下事项必须在数据库 Migration 或关键链路代码开始前落实：

1. **统一约束字段名称。** 后续实现只使用 `SeatId + ReservationDate + TimeSlot` 和 `ExperienceUserId + ReservationDate + TimeSlot`，不得使用不存在的 `Date`、`UserId` 字段。
2. **固定枚举数据库值。** `TimeSlot`、`StoredStatus`、`CancelledBy` 必须显式映射为 `tinyint`，并保持文档中的 1/2/3 数值不变。
3. **固定过滤索引配置。** 两个唯一索引都必须包含 SQL Server Filter：`[StoredStatus] = 1`，Migration 生成后需人工检查索引 SQL。
4. **固定种子数据幂等规则。** 运行时初始化器只在目标表为空或不存在指定种子标识时插入数据，不得每次启动追加预约。
5. **固定动态展示状态查询。** 下一步需写出 EF Core 可翻译的 Reserved、Cancelled、Completed 筛选表达式，并统一使用注入的 `TimeProvider`。

## D. 建议优化项

1. 将两个过滤唯一索引名称中的 `User` 改为 `ExperienceUser`，使命名更贴近正式字段，但不影响数据库功能。
2. 在 EF Core Configuration 中为三个枚举集中配置转换，避免每个查询手写数字常量。
3. 为 CHECK 约束和过滤唯一索引写 4 个最小数据库测试：座位冲突、用户冲突、取消后重约、非法状态拒绝。
4. 管理端动态状态筛选应直接在数据库查询中完成，不要先 `ToListAsync()` 再过滤。
5. 种子预约应选择不同座位或时段，确保不会触发两条过滤唯一索引。
6. README 后续增加“删除 LocalDB 并重新初始化演示数据”的明确命令，便于课堂验收复位。
7. 如果后续增加多个管理员，再评估 `CancelledByAdminId` 或预约操作日志；本期不要提前扩表。

## E. 是否可以进入下一步“关键链路详细设计”

**可以有条件进入。**

实体、关系、状态、查询、统计和一致性约束已经足够支撑关键链路设计，不需要重新建模。下一步应优先详细设计：

1. 创建预约的预检查、写入、唯一索引异常转换和返回错误代码。
2. 用户取消与管理员取消的时间边界、RowVersion 并发处理和状态回显。
3. 动态展示状态的统一计算及数据库筛选表达式。
4. 管理端日期、状态、区域、用户组合查询。
5. 幂等种子初始化和数据库重建流程。

完成 C 节事项后，可直接按当前 4 表结构进入 EF Core Entity、Configuration、Migration 和 Service 开发，不需要增加默认 6 实体中的其他表。
