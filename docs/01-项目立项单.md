# 图书馆座位预约系统项目立项单

## 1. 项目基本信息

| 项目项 | 内容 |
| --- | --- |
| 项目名称 | 图书馆座位预约系统 |
| 项目阶段 | 课程第三阶段：模拟入职后独立负责第一个小项目 |
| 项目类型 | Web 管理系统 / 课程实践项目 |
| 技术栈 | ASP.NET Core MVC (.NET 8) + Razor + SQL Server LocalDB + EF Core + Bootstrap |
| 主要使用者 | 学生用户、图书馆管理员 |
| 本期目标 | 完成一个可运行、可演示、可提交仓库链接的 MVP 版本 |
| 最终提交方式 | GitHub 或 Gitee 仓库链接 |

## 2. 项目背景

图书馆自习座位在学习高峰期容易出现“找座难、占座混乱、管理员统计不方便”等问题。  
本项目模拟一个学校图书馆座位预约场景，让学生可以在线查看座位并提交预约，让管理员可以管理座位和预约记录。

本期不是做一个完整商业系统，而是做一个课堂项目 MVP。重点是把一个小项目从立项、需求、设计、开发、测试到提交跑通，让实习生熟悉真实项目的基本工作节奏。

## 3. 目标用户

| 用户类型 | 主要需求 |
| --- | --- |
| 学生用户 | 查看座位、选择座位、提交预约、查看自己的预约、取消预约 |
| 图书馆管理员 | 登录后台、查看预约记录、维护座位信息、查看基础统计 |
| 课程验收老师 | 通过页面操作判断系统是否完整、流程是否跑通、代码是否可运行 |

## 4. 项目目标

1. 完成一个基于 ASP.NET Core MVC 的图书馆座位预约系统。
2. 实现用户端和管理端两条基本业务线。
3. 使用 SQL Server LocalDB 和 EF Core 保存座位、预约、管理员等核心数据。
4. 页面使用 Razor + Bootstrap，保证界面清楚、操作简单、便于课堂演示。
5. 控制功能范围，不做复杂账号体系和额外扩展，优先保证主流程稳定。
6. 最终提交可运行项目仓库链接，方便老师或同学下载、运行、检查。

## 5. 本期必须完成内容

### 用户端

1. 用户首页：展示系统入口、当前可预约座位概况、进入座位列表。
2. 座位列表：展示座位编号、区域、状态和备注信息。
3. 座位详情：展示单个座位信息，并提供预约入口。
4. 预约提交：用户选择座位和预约时间段后提交预约。
5. 我的预约：查看当前体验用户的预约记录。
6. 取消预约：用户可以取消自己未完成或未过期的预约。
7. 体验账号切换或轻量模拟登录：用于课堂演示不同用户的数据隔离效果。

### 管理端

1. 管理员登录：使用轻量 Session / Cookie 登录，不接入复杂权限系统。
2. 预约管理：查看预约列表，按状态进行基本管理。
3. 座位管理：新增、编辑、停用或启用座位。
4. 统计页：展示座位总数、可用座位数、预约数量等基础统计信息。

### 数据与工程

1. 使用 EF Core 建立核心实体与数据库表。
2. 使用 SQL Server LocalDB 作为本地开发数据库。
3. 保留必要的种子数据，方便第一次运行后直接演示。
4. 项目结构保持清晰，便于后续写需求分析、数据库设计和测试说明。

## 6. 本期暂不开发内容

1. 不做支付、收费、押金相关功能。
2. 不做扫码签到、扫码离座功能。
3. 不做短信、邮件、微信、站内消息推送。
4. 不做复杂账号注册、找回密码、第三方登录。
5. 不做多角色细粒度权限控制。
6. 不做多馆区、多楼层联动调度。
7. 不做移动端 App 或小程序。
8. 不做复杂报表导出和大屏展示。
9. 不做真实生产环境部署，只保证本地运行和课堂演示。

## 7. 页面规模与页面分布

本期页面固定为 9 页：用户端 5 页、管理端 4 页，避免页面失控。

| 模块 | 页面 | 说明 |
| --- | --- | --- |
| 用户端 | 用户首页 | 系统入口和预约引导 |
| 用户端 | 座位列表 | 查看全部座位和状态 |
| 用户端 | 座位详情 | 查看单个座位信息 |
| 用户端 | 预约提交 | 提交座位预约 |
| 用户端 | 我的预约 | 查看当前用户预约 |
| 管理端 | 管理员登录 | 后台登录入口 |
| 管理端 | 预约管理 | 查看和处理预约 |
| 管理端 | 座位管理 | 维护座位基础信息 |
| 管理端 | 统计页 | 展示基础统计数据 |

取消预约固定为“我的预约”页面中的按钮和确认交互，不单独拆成业务页面；座位新增、编辑和启停固定在“座位管理页”同页完成。

## 8. 关键业务主流程

### 学生预约流程

1. 学生进入用户首页。
2. 选择或切换一个体验账号。
3. 进入座位列表，查看座位状态。
4. 点击某个可预约座位进入座位详情。
5. 选择预约日期和时间段。
6. 提交预约。
7. 系统保存预约记录，并在“我的预约”中展示。
8. 学生可以在“我的预约”中取消未完成预约。

### 管理员管理流程

1. 管理员进入后台登录页。
2. 输入管理员账号和密码完成登录。
3. 进入预约管理页，查看全部预约记录。
4. 进入座位管理页，新增或编辑座位信息。
5. 管理员可以停用暂不可用座位。
6. 管理员进入统计页，查看座位和预约的基础统计。

### 核心限制流程

1. 已停用座位不能被用户预约。
2. 已被预约的同一座位在同一时间段不能重复预约。
3. 用户只能取消自己的预约。
4. 未登录管理员不能进入管理端页面。

## 9. 成功标准

本项目验收时，至少能完成以下演示动作：

1. 运行项目后，浏览器可以打开用户首页。
2. 在用户端可以切换体验用户。
3. 在座位列表中可以看到多条座位数据。
4. 点击一个可用座位，可以进入座位详情页。
5. 选择时间段并提交预约后，页面提示预约成功。
6. 进入“我的预约”，可以看到刚才提交的预约记录。
7. 点击取消预约后，该预约状态发生变化或从有效预约中移除。
8. 打开管理员登录页，输入管理员账号后可以进入后台。
9. 在预约管理页可以看到用户提交的预约记录。
10. 在座位管理页可以新增或编辑一个座位。
11. 将某个座位设置为停用后，用户端不能继续预约该座位。
12. 统计页可以展示座位总数、可用座位数、预约总数等数据。
13. 项目代码提交到 GitHub 或 Gitee，仓库中包含运行说明。

## 10. 风险与假设

| 风险 | 可能影响 | 应对方式 |
| --- | --- | --- |
| 时间不够 | 页面做不完，流程不完整 | 优先保证用户预约主流程和管理员查看流程，弱化装饰性页面 |
| 页面数量失控 | 开发时间被大量页面消耗 | 页面固定为用户端 5 页、管理端 4 页；取消预约只做页内按钮操作 |
| 数据库设计过重 | 表太多、关系太复杂，调试困难 | 只保留用户、管理员、座位、预约等核心数据 |
| AI 生成代码不稳定 | 可能出现编译错误或逻辑不一致 | 每完成一个功能就运行一次，及时修正，不堆到最后 |
| LocalDB 环境差异 | 同学或老师电脑可能无法直接连接数据库 | README 中写清楚数据库连接和迁移命令 |
| 登录功能做复杂 | 占用过多时间，影响核心预约功能 | 用户端用体验账号切换，管理端用轻量 Session / Cookie |
| Bootstrap 页面不统一 | 演示时观感杂乱 | 使用统一布局页和基础组件，不追求复杂视觉效果 |

本期假设：

1. 系统主要用于课堂演示，不承载真实高并发访问。
2. 预约时间段可以先使用固定选项，不做复杂日历控件。
3. 用户端体验账号可以由系统预置，不要求真实注册。
4. 管理员账号可以使用种子数据或配置中的固定账号。
