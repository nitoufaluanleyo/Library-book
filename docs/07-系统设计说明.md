# 图书馆座位预约系统系统设计说明

## 文档信息

| 项目 | 内容 |
| --- | --- |
| 项目名称 | 图书馆座位预约系统 |
| 项目类型 | 单人课堂 MVC 项目 |
| 技术栈 | ASP.NET Core MVC (.NET 8) + Razor + EF Core + SQL Server LocalDB + Bootstrap |
| 正式原型基线 | `prototype/static-v2/` |
| 页面范围 | 用户端 5 页 + 管理端 4 页，共 9 页 |
| 下一步 | 数据库设计 |

本说明负责回答“代码放在哪里、每层负责什么、页面请求经过哪些代码”。本期采用单解决方案、单 Web 项目的分层目录，不拆微服务，不为课堂项目增加不必要的独立类库。

## 1. 系统设计目标

1. 将已确认的 9 页静态原型转换为可运行的 ASP.NET Core MVC 页面，不扩大页面和功能范围。
2. 让 Controller 只处理 HTTP 请求，业务规则集中在 Service，数据库访问集中在 DataAccess。
3. 使用 ViewModel 作为页面输入输出，禁止 Razor View 直接绑定 EF Core Entity。
4. 用统一服务实现日期、时段、冲突、取消和展示状态规则，避免用户端与管理端各写一套判断。
5. 使用 Session 保存当前体验用户，使用 Cookie 保护管理端，两类身份互不混用。
6. 保持项目结构简单，使一名实习生能在课堂周期内完成、调试和演示。
7. 为下一步数据库设计明确 4 个核心实体、枚举、关系和约束落点。

### 1.1 已冻结口径

| 项目 | 系统设计口径 |
| --- | --- |
| 正式视觉与假数据参考 | 使用 `static-v2`；`static-v1` 只作第一版对照 |
| 体验用户 | 2 个预置体验用户 |
| 管理员 | 1 个预置管理员账号 |
| 初始座位 | 8 个，其中 7 个启用、1 个停用 |
| 初始预约 | 3 条，用于展示已预约、已完成、已取消 |
| 可预约日期 | 服务器当天至当天 + 6 天，共 7 天 |
| 固定时段 | 上午 08:00-12:00、下午 14:00-18:00、晚上 18:30-21:30 |
| 数据库存储状态 | 已预约、已取消 |
| 页面展示状态 | 已预约、已取消、已完成；已完成根据结束时间动态计算 |
| 页面数量 | 9 页，不新增成功页、确认页、管理首页或独立编辑页 |
| 统计范围 | 只实现 PRD 规定的 5 个数字指标；趋势和分布图暂不接真实统计逻辑 |

## 2. 技术栈说明

| 技术 | 本项目用途 | 使用边界 |
| --- | --- | --- |
| .NET 8 | 应用运行时 | 使用当前 LTS，不混用旧版模板 |
| ASP.NET Core MVC | 路由、Controller、模型绑定、过滤器 | 本期不建设 Web API 和前后端分离工程 |
| Razor | 服务端页面渲染、Layout、Partial、Tag Helper | 不引入 React、Vue 等前端框架 |
| EF Core 8 | 实体映射、查询、迁移和事务 | Service 通过 `AppDbContext` 访问，不再套通用 Repository |
| SQL Server LocalDB | 本地课堂演示数据库 | 单机开发，不考虑集群、读写分离和云数据库 |
| Bootstrap | 布局、表单、表格、Modal、Offcanvas | 复用原型样式，不把 `prototype.js` 当生产业务代码 |
| Session | 保存当前体验用户编号 | 只保存必要标识，不保存完整用户对象和预约数据 |
| Cookie Authentication | 管理员登录状态 | 使用独立管理端认证方案，不引入 ASP.NET Core Identity |
| Data Annotations | ViewModel 输入校验 | 复杂业务规则仍由 Service 校验 |
| ILogger | 记录异常和关键失败信息 | 日志不记录明文密码和敏感 Cookie |

### 2.1 设计取舍

1. **单 Web 项目。** 当前只有 9 页和 4 个核心实体，拆成多个类库会增加引用和部署成本。
2. **不做通用 Repository。** EF Core 的 `DbContext` 已承担工作单元职责；本项目在 DataAccess 中集中 DbContext、实体配置、迁移和种子数据即可。
3. **Service 作为业务规则入口。** Controller 不直接查询或修改 `AppDbContext`，这样冲突和取消规则可以被用户端、管理端共同复用。
4. **服务端渲染优先。** 筛选、表单提交和状态反馈采用 MVC 请求；Bootstrap JavaScript 只负责 Modal、Offcanvas 等界面交互。

### 3.1 目录使用规则

1. `Areas/Admin` 只放管理端 Controller 和 View，管理端统一使用 `/Admin/...` 路由。
2. `Controllers` 只放用户端入口，不放数据库查询代码。
3. `Services` 放业务规则和页面数据组装；接口与实现分开，便于测试和依赖注入。
4. `DataAccess` 放 EF Core 基础设施，不放页面 ViewModel。
5. `Models/Entities` 只描述数据库持久化字段和导航属性。
6. `ViewModels` 按页面或功能分目录，类名直接对应页面用途。
7. `Views/Shared` 放用户端共享布局与小型 Partial；管理布局放在 Admin Area 内。
8. `wwwroot/css/site.css` 从正式原型 `static-v2/assets/styles.css` 整理迁移；不直接复制原型状态脚本。
9. `tests` 首先覆盖 ReservationService 的冲突、取消和展示状态规则，不追求一次写全所有页面测试。

## 4. 分层说明

### 4.1 请求依赖方向

```text
浏览器
  -> Controller
  -> Service
  -> DataAccess (AppDbContext)
  -> SQL Server LocalDB

Service -> Entity / Enum
Controller <-> ViewModel <-> Razor View
```

依赖只按箭头方向流动。View 不访问 Service 或 DbContext，Controller 不直接写数据库，Entity 不引用 ViewModel。

### 4.2 Controller 层

**放置位置：** `Controllers/`、`Areas/Admin/Controllers/`

**负责：**

1. 接收路由、查询参数和表单数据。
2. 触发模型绑定和基础输入校验。
3. 从 Service 获取 ViewModel 或操作结果。
4. 根据结果返回 View、Redirect、NotFound 或 Forbid。
5. 写入一次性成功提示 `TempData`，采用 POST-Redirect-GET 防止重复提交。

**不负责：**

1. 不直接注入或调用 `AppDbContext`。
2. 不在 Action 中手写座位冲突、用户冲突和时间判断。
3. 不相信表单提交的体验用户编号或管理员编号。
4. 不把 Entity 直接传给 Razor View。

### 4.3 Service 层

**放置位置：** `Services/`

**负责：**

1. 根据页面需求查询数据并组装 ViewModel。
2. 校验座位启停、日期范围、时段是否开始、座位冲突和用户冲突。
3. 创建预约、取消预约、维护座位和计算统计指标。
4. 统一计算预约展示状态，避免页面自行判断“已完成”。
5. 返回 `ServiceResult`，包含是否成功、错误代码、用户提示和必要数据。
6. 对多个写操作使用 EF Core 事务，保证校验与写入尽量在同一业务过程完成。

**建议服务：**

| 服务 | 核心职责 |
| --- | --- |
| `ICurrentExperienceUserService` | 读取、校验和切换 Session 中的体验用户 |
| `ISeatService` | 座位列表、详情、管理员新增和编辑 |
| `IReservationService` | 创建页日期/时段可用性、创建预约、我的预约、管理员列表、两类取消和状态计算 |
| `IAdminAuthService` | 校验预置管理员账号和密码哈希，生成登录所需 Claims 数据 |
| `IStatisticsService` | 查询 5 个固定数字指标 |

### 4.4 DataAccess 层

**放置位置：** `DataAccess/`

**负责：**

1. `AppDbContext` 暴露 `DbSet<ExperienceUser>`、`DbSet<Admin>`、`DbSet<Seat>`、`DbSet<Reservation>`。
2. `Configurations` 使用 Fluent API 定义表名、字段长度、必填项、索引、唯一约束和关系。
3. `Migrations` 保存数据库结构变更记录。
4. `DbSeeder` 创建课堂演示所需的 2 个体验用户、1 个管理员、8 个座位和 3 条相对日期预约。
5. 查询操作默认使用 `AsNoTracking()`；需要更新的对象才保持跟踪。

**约束：**

1. 本期不增加通用 `IRepository<T>` 和自定义 UnitOfWork。
2. Controller 不直接访问 DataAccess。
3. 数据库唯一约束必须作为最终防线，不能只依赖页面或 Service 的预检查。

### 4.5 ViewModel 层

**放置位置：** `ViewModels/`

**负责：**

1. 只携带页面需要展示或提交的字段。
2. 使用 `[Required]`、`[StringLength]` 等属性完成表单基础校验。
3. 为列表页包含筛选条件、结果列表、空状态和分页外的必要信息。
4. 将 Entity 的存储状态转换为适合页面显示的文本、标签样式和可操作标志。

**示例：**

| ViewModel | 关键字段 |
| --- | --- |
| `HomeIndexViewModel` | 当前体验用户、用户选项、座位总数、启用座位数 |
| `SeatListViewModel` | 区域筛选、座位卡片列表、是否为空 |
| `SeatDetailsViewModel` | 座位信息、启停状态、是否显示预约入口 |
| `ReservationCreateViewModel` | SeatId、座位摘要、当前体验用户、选中日期/时段、7 天日期、3 个时段及座位/用户占用状态、错误信息 |
| `MyReservationsViewModel` | 当前用户、待使用列表、历史列表、可取消标志 |
| `AdminSeatListViewModel` | 筛选、座位列表、同页 `SeatEditViewModel` |
| `StatisticsViewModel` | 座位总数、启用数、停用数、预约总数、今日非取消预约数 |

ViewModel 可以包含页面专用的子项类型，例如 `SeatCardItemViewModel`、`TimeSlotOptionViewModel`，不要为了省文件而使用匿名对象。

### 4.6 Entity 层

**放置位置：** `Models/Entities/`、`Models/Enums/`

**负责：**

1. 表示数据库需要持久化的数据和实体关系。
2. 保存简单且稳定的实体自身约束，例如默认值和导航属性。
3. 不包含 Session、Cookie、HTTP、TempData 和页面样式逻辑。

下一步数据库设计围绕以下实体展开：

| Entity | 主要用途 | 核心关系 |
| --- | --- | --- |
| `ExperienceUser` | 预置学生体验账号 | 一对多关联 Reservation |
| `Admin` | 预置管理员账号和密码哈希 | 取消方通过业务字段记录，不建立复杂权限模型 |
| `Seat` | 座位编号、区域、备注、启停状态 | 一对多关联 Reservation |
| `Reservation` | 用户在某座位、日期、时段的预约记录 | 多对一关联 ExperienceUser 和 Seat |

## 7. 主链路工程落点

### 7.1 用户预约成功链路

```text
U-P01 HomeController.Index
  -> CurrentExperienceUserService 获取当前体验用户
  -> U-P02 SeatsController.Index 获取座位列表
  -> U-P03 SeatsController.Details 只获取座位详情和启停状态
  -> U-P04 ReservationsController.Create GET
       1. CurrentExperienceUserService 获取当前体验用户
       2. ReservationService.BuildCreatePageAsync 组装 7 天、3 时段
       3. 同时标记座位占用和当前用户同一时段占用
  -> U-P04 ReservationsController.Create POST
  -> ReservationService.CreateAsync
       1. 从 Session 对应服务获取用户
       2. 校验日期和时段
       3. 重新查询 Seat 并检查启用状态
       4. 检查座位冲突
       5. 检查用户冲突
       6. 新增 Reservation 并 SaveChanges
  -> TempData 写入成功提示
  -> Redirect 到 U-P05 ReservationsController.Mine
```

Controller 只接收 `seatId/date/slot`，用户编号必须从 Session 服务端读取。

### 7.2 用户取消链路

```text
U-P05 Cancel POST
  -> ReservationService.CancelByUserAsync(reservationId, currentUserId, now)
  -> 校验记录存在、归属当前用户、状态为已预约、尚未开始
  -> 更新 Status、CancelledAt、CancellationActor
  -> SaveChanges
  -> Redirect 回 U-P05 并显示结果
```

不能根据页面上是否显示取消按钮决定权限，POST 时必须重新校验。

### 7.3 管理员取消链路

```text
A-P01 登录成功并生成 AdminCookie
  -> A-P02 Cancel POST 经过 [Authorize]
  -> ReservationService.CancelByAdminAsync(reservationId, adminId, now)
  -> 校验记录状态为已预约且尚未结束
  -> 更新取消信息
  -> Redirect 回筛选后的 A-P02
```

用户取消按“开始时间”限制，管理员取消按“结束时间”限制，两者必须使用不同 Service 方法，不能共用一个模糊布尔参数。

### 7.4 管理员维护座位链路

```text
A-P03 GET Index(editId)
  -> SeatService 获取列表和编辑对象
  -> 同页 Offcanvas 显示 SeatEditViewModel
A-P03 POST Save
  -> 基础 ModelState 校验
  -> SeatService.SaveAsync
  -> 校验座位编号唯一
  -> 新增或更新允许字段
  -> Redirect 回 A-P03 并显示成功提示
```

本期不提供物理删除。停用座位只阻止新预约，不自动取消已有预约。

### 7.5 统计链路

```text
A-P04 StatsController.Index
  -> StatisticsService.GetOverviewAsync(today)
  -> Seat 与 Reservation 分别执行聚合查询
  -> StatisticsViewModel
  -> Razor 渲染 5 个数字指标
```

统计页不加载趋势和导出数据。若保留原型中的趋势/分布区域，应显示为非功能占位或在开发版中移除，不能为此增加数据库查询范围。

## 8. 异常处理原则

### 8.1 可预期业务错误

以下情况不抛系统异常，由 Service 返回失败的 `ServiceResult`：

1. 座位不存在或已停用。
2. 日期超出 7 天范围。
3. 当天时段已经开始。
4. 座位在同日期同一时段已被预约。
5. 当前用户在同日期同一时段已有预约。
6. 预约不存在、归属不符或已不能取消。
7. 座位编号为空或重复。
8. 管理员账号或密码错误。

Controller 将错误加入 `ModelState` 或 `TempData`，返回原页或重定向到安全页面。提示应说明用户下一步可以修改什么，不显示数据库和堆栈信息。

### 8.2 HTTP 结果原则

| 场景 | 处理方式 |
| --- | --- |
| 表单字段不合法 | 返回当前 View，显示字段级错误 |
| 业务冲突 | 返回当前 View 或重定向，显示业务提示 |
| 用户端资源不存在 | 返回 `NotFound()`，由统一错误视图展示 |
| 未登录访问管理端 | Cookie 中间件重定向到 `/admin/login` |
| 已登录但无权操作 | 返回 `Forbid()` 或安全提示，不执行写入 |
| 非预期异常 | 全局异常处理中间件记录日志并显示通用错误页 |

统一错误页属于框架级兜底页面，不计入 9 个核心业务页面。

### 8.3 写操作原则

1. 所有 POST Action 使用 `[ValidateAntiForgeryToken]`。
2. 成功写入后使用 Redirect，避免刷新重复提交。
3. 数据库保存前后都不能信任页面按钮状态，必须重新读取数据库状态。
4. 捕获唯一约束或并发冲突时，转换为可理解的业务提示。
5. 日志记录操作类型和记录编号，不记录管理员明文密码。
6. 开发环境可显示详细错误；非开发环境只显示统一提示。

## 9. Session / Cookie 使用原则

### 9.1 体验用户 Session

| 项目 | 设计 |
| --- | --- |
| Session Key | `CurrentExperienceUserId` |
| 保存内容 | 只保存体验用户主键 |
| 默认行为 | 首次访问时选择第一个启用的种子体验用户 |
| 切换方式 | POST `/experience-account/switch`，带防伪令牌 |
| 有效校验 | 每次读取后确认用户仍存在且启用；无效时回退到默认体验用户 |
| 建议超时 | 30 分钟空闲超时 |
| 安全边界 | 任何预约查询、创建和取消都从 Session 取用户编号，不接受表单中的 UserId |

Session 只解决课堂演示中的“当前体验用户”上下文，不代表真实身份认证，也不能用于访问管理端。

### 9.2 管理员 Cookie

| 项目 | 设计 |
| --- | --- |
| Scheme | `AdminCookie` |
| Claims | `NameIdentifier` 保存 AdminId，`Name` 保存显示名 |
| 登录页 | `/admin/login` |
| 拒绝页 | 可回登录页并显示无权限提示 |
| Cookie 属性 | `HttpOnly=true`、`SameSite=Lax`、生产环境要求 Secure |
| 建议有效期 | 2 小时，可启用滑动过期 |
| 密码处理 | 数据库存密码哈希，不保存或比较明文密码 |
| 访问保护 | Admin Area Controller 统一使用 `[Authorize(AuthenticationSchemes = "AdminCookie")]` |

`AuthController.Login` 标记允许匿名访问。退出必须使用 POST，执行 `SignOutAsync("AdminCookie")`，不能只跳转页面。

### 9.3 两类状态隔离

1. Session 中存在体验用户不代表管理员已登录。
2. 管理员 Cookie 不改变当前体验用户 Session。
3. 用户端永远不从管理员 Claims 推断体验用户。
4. 管理端写操作从 Cookie Claims 读取管理员编号，用于记录取消方。
5. 不在 URL、隐藏字段或 JavaScript 中保存可直接信任的身份编号。

## 11. 进入数据库设计的确认结论

本系统的页面范围、分层职责、核心模块、路由、身份状态和异常处理已经明确，可以进入下一步《数据库设计》。数据库设计必须围绕 `ExperienceUser`、`Admin`、`Seat`、`Reservation` 四个实体展开，并优先保证预约冲突、用户数据隔离、取消状态和统计口径能够被服务端可靠实现。
