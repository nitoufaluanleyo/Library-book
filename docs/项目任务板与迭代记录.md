# 图书馆座位预约系统项目任务板与迭代记录

## 1. 使用说明

本文件使用 Markdown 模拟个人任务板。每轮开始时选择“待开发”卡片进入“进行中”，实现完成后进入“待验证”，通过验收后进入“已完成”。

Sprint 1-4 都是**主 Sprint，可多轮推进**。一个主 Sprint 可以有第 1 轮、第 2 轮、修复轮和验收轮，不以一次对话、一次提交或一天作为结束条件。

## 2. 任务卡编号规则

| 编号前缀 | 对应阶段 | 示例 |
| --- | --- | --- |
| `T12-xx` | Sprint 0：开发起步与项目骨架 | `T12-02` 创建解决方案 |
| `T13-xx` | Sprint 1：体验用户与座位浏览 | `T13-03` 完成座位列表 |
| `T14-xx` | Sprint 2：用户预约闭环 | `T14-03` 完成预约写入 |
| `T15-xx` | Sprint 3：管理端闭环 | `T15-04` 完成管理员取消 |
| `T16-xx` | Sprint 4：统计、联调与交付 | `T16-01` 完成五项统计 |

新增卡片使用当前 Sprint 下一个空闲编号，不复用已关闭编号。

## 3. 状态定义

| 状态 | 进入条件 | 离开条件 |
| --- | --- | --- |
| 待开发 | 范围和验收标准已明确 | 开始实际工作 |
| 进行中 | 正在编码或编写当前产物 | 完成自检并提交验证 |
| 待验证 | 产物已完成但尚未验收 | 验证通过或退回修复 |
| 已完成 | 验收通过并更新记录 | 保持关闭；回归失败时新建修复卡 |

## 17. 任务板维护规则

1. 卡片开始时立即移到“进行中”，不允许事后补记。
2. 代码写完但未 build/测试时只能进入“待验证”。
3. 只有验收标准全部满足且“本轮是否验证通过”为“是”时，才能进入“已完成”。
4. 发现超出范围的需求不直接开发，记录到暂缓清单或新建待评审卡片。
5. 每轮结束更新任务统计、本轮新增完成卡片、完成率和是否允许进入下一阶段。
6. 完成率按“已完成卡片数 / 当前 Sprint 卡片总数”计算，不按主观工作量估算。
7. 每个重要节点自动执行 Git commit；推送前检查不包含相邻项目或本地敏感文件。
