## 1. 项目概述

### 1.1 项目背景

校园图书馆在学习高峰期容易出现到馆后找不到座位、同一座位重复占用、管理员难以快速查看预约情况等问题。本项目通过一个轻量 Web 系统，让学生体验用户在到馆前查看座位并预约，让管理员维护座位和预约数据。

本项目不是商业化图书馆平台，而是课程第三阶段的单人 MVP。重点是完成用户预约和管理员管理两条可运行、可演示、可检查的业务闭环。

### 1.2 产品定位

一个面向单馆区课堂演示的图书馆座位预约系统，使用预置体验用户和预置管理员账号，不接入真实校园身份系统。

### 1.3 已统一的关键口径

| 事项 | PRD Lite 最终口径 |
| --- | --- |
| 预约日期 | 从服务器当天起，含当天共 7 个自然日，即今天至今天 + 6 天 |
| 预约时段 | 上午 08:00-12:00、下午 14:00-18:00、晚上 18:30-21:30 |
| 当天过期时段 | 当前时间大于或等于时段开始时间后，学生不能再预约该时段 |
| 座位状态 | 启用、停用；表示座位能否参与预约，不表示某个时段是否已被占用 |
| 时段可用性 | 用户在座位详情页选择日期后，根据该座位已有预约动态判断 |
| 数据库存储状态 | 预约只存“已预约、已取消” |
| 页面展示状态 | 已预约、已取消、已完成；“已完成”由当前时间超过或等于结束时间动态计算 |
| 操作反馈 | 成功提示、失败原因和服务端校验反馈属于 MVP 必做能力 |
| 非 MVP 功能 | 统一记为“暂缓”，并在“暂不开发内容”中汇总，不再使用“删除”作为另一种状态 |

## 2. 项目目标

1. 完成学生从选择体验账号、查看座位、提交预约到查看和取消预约的完整流程。
2. 完成管理员登录、预约管理、座位管理和基础统计流程。
3. 对座位冲突、用户冲突、过期时段、停用座位和越权取消进行服务端校验。
4. 使用 4 类核心数据对象和约 9 个主要页面控制课堂项目工作量。
5. 提供可初始化的演示数据和明确的 README，使项目能在本地启动和验收。
6. 最终将可运行代码提交到 GitHub 或 Gitee，提供仓库链接。

项目成功的最低结果不是“页面做出来”，而是预约、冲突拦截、数据隔离、取消、后台保护和数据变化都能通过实际操作演示。

## 5. 用户端核心功能

### U01 体验账号切换

**功能说明：** 用户端顶部或首页提供 2 个预置体验账号。首次访问默认选择第一个启用账号；切换后，服务器将当前体验用户编号写入 Session，后续查询和写入均以 Session 中的用户为准，不接受前端提交任意用户编号。

**最小验收标准：**

1. 页面能显示当前体验用户名称，并可在两个预置账号间切换。
2. 切换后刷新页面，当前体验用户在 Session 有效期内保持不变。
3. “我的预约”只显示当前体验用户的数据。
4. 用户即使修改请求参数，也不能查看或取消其他用户的预约。

### U02 用户首页

**功能说明：** 首页显示当前体验用户、座位总数、启用座位数和进入座位列表、“我的预约”的入口。启用座位数只表示可参与预约的座位数量，不代表指定日期和时段一定空闲。

**最小验收标准：**

1. 项目启动后可以打开首页，页面无服务器错误。
2. 首页显示当前体验用户、座位总数和启用座位数。
3. 首页可进入座位列表和“我的预约”。
4. 没有座位数据时显示明确空状态，不显示错误页面。

### U03 座位列表与区域筛选

**功能说明：** 列表显示座位编号、区域、备注和启用状态，支持按区域进行单条件筛选。列表不直接计算所有日期和时段的可用性，具体时段可用性在座位详情页判断。

**最小验收标准：**

1. 列表能显示 8 条座位种子数据，其中 7 条启用、1 条停用。
2. 用户选择区域后，只显示对应区域的座位；清除筛选后恢复全部座位。
3. 启用和停用座位有清楚的文字或样式区分。
4. 点击启用座位可以进入详情；停用座位不能进入可提交预约的状态。

### U04 座位详情与预约提交

**功能说明：** 详情页显示座位编号、区域、备注和启停状态。用户选择含当天共 7 个自然日内的日期后，页面展示 3 个固定时段及其可选状态。点击预约后，服务器再次校验日期、时段、座位状态和冲突，再决定是否写入。

**最小验收标准：**

1. 启用座位详情页可选择有效日期和可用时段。
2. 过去日期、今天已经开始的时段、7 天范围外日期均不能预约。
3. 停用座位提交预约时，服务端拒绝并显示明确原因。
4. 合法提交后只新增 1 条预约，提示预约成功，并可进入“我的预约”查看。
5. 重复点击提交或重复发送同一请求时，不应无提示地产生重复预约。

### U05 预约冲突校验与操作反馈

**功能说明：** 提交预约时必须同时检查座位冲突和用户冲突。所有新增、取消等写操作都要显示成功或失败结果，失败时说明可理解的原因。

**最小验收标准：**

1. 同一座位、同一日期、同一时段已有“已预约”记录时，第二次提交失败。
2. 同一用户、同一日期、同一时段已有其他座位预约时，第二次提交失败。
3. “已取消”记录不占用座位，也不阻止同一用户重新预约。
4. 冲突校验在服务端执行，不能通过修改按钮状态或请求参数绕过。
5. 失败后不写入无效预约，并在页面显示对应失败原因。

### U06 我的预约与用户取消

**功能说明：** “我的预约”按预约日期和时段倒序展示当前体验用户的记录，包括座位、日期、时段和展示状态。用户只能取消尚未开始的有效预约，取消操作在当前页面完成，不新增独立取消页面。

**最小验收标准：**

1. 页面只显示当前体验用户的预约，包含已预约、已取消、已完成状态。
2. 只有“已预约”且当前时间早于开始时间的记录显示取消按钮。
3. 用户确认取消后，数据库状态改为“已取消”，并记录取消时间和取消方。
4. 取消后对应座位时段可被其他用户重新预约。
5. 用户取消他人记录、已取消记录或已开始记录时，服务端拒绝并给出提示。

## 6. 管理端核心功能

### A01 管理员登录、退出与访问保护

**功能说明：** 使用预置管理员账号完成轻量 Cookie 登录，不接入 ASP.NET Core Identity 和复杂角色系统。管理端页面统一检查管理员登录状态。

**最小验收标准：**

1. 正确账号和密码可以登录并进入 A-P04 统计页。
2. 错误账号或密码不能登录，并显示统一错误提示，不指出具体错误字段。
3. 未登录访问任一管理端页面时跳转到管理员登录页。
4. 点击退出后登录状态失效，再次访问管理端需要重新登录。

### A02 预约管理

**功能说明：** 管理员可查看全部体验用户的预约，按预约日期和展示状态进行基础筛选，并取消尚未结束的有效预约。不做审批、确认、改签、分页和批量操作。

**最小验收标准：**

1. 列表显示体验用户、座位、日期、时段、展示状态和创建时间。
2. 可按单个日期、单个展示状态筛选，清除条件后恢复全部记录。
3. 管理员可取消数据库状态为“已预约”且当前时间早于结束时间的记录。
4. 管理员取消后记录状态为“已取消”，并记录取消时间和取消方为管理员。
5. 已取消或已完成记录不能再次取消，服务端拒绝重复操作。

### A03 座位管理

**功能说明：** 管理员通过座位列表和新增/编辑共享表单维护座位编号、区域、备注和启停状态。本期不做物理删除，不自动取消停用前已存在的预约。

**最小验收标准：**

1. 管理员可以新增座位，座位编号必填且不能重复。
2. 管理员可以编辑座位区域、备注和启停状态。
3. 座位停用后，用户端不能再为该座位创建新预约。
4. 停用座位已有的历史或有效预约仍保留，管理员可在预约管理页按规则处理。
5. 页面不提供物理删除入口。

### A04 基础统计

**功能说明：** 统计页只显示数字，不做图表、趋势和导出。

| 指标 | 计算口径 |
| --- | --- |
| 座位总数 | 全部座位记录数量 |
| 启用座位数 | `IsEnabled = true` 的座位数量 |
| 停用座位数 | `IsEnabled = false` 的座位数量 |
| 预约总数 | 全部预约记录数量，包含已取消和展示为已完成的记录 |
| 今日非取消预约数 | 预约日期为服务器当天且数据库状态不是“已取消”的记录数量，包含当天动态展示为“已完成”的记录 |

**最小验收标准：**

1. 统计页显示以上 5 个指标，空数据库时显示 0。
2. 新增座位后座位总数和对应启停数量正确变化。
3. 新增预约后预约总数正确变化；取消预约后预约总数不减少。
4. 统计值来自数据库查询，不使用页面写死数字。

## 8. 核心业务流程

### 8.1 学生预约成功流程

1. 用户进入 U-P01，确认或切换当前体验账号。
2. 用户进入 U-P02，按区域查看座位并选择一个启用座位。
3. 用户进入 U-P03 查看座位信息，并进入 U-P04。
4. 用户在 U-P04 选择有效日期，系统根据当前时间、座位状态、当前用户和已有预约显示 3 个时段的可选状态。
5. 用户选择可用时段并提交。
6. 服务器从 Session 读取当前用户，再次检查日期范围、时段、座位状态、座位冲突和用户冲突。
7. 校验通过后创建“已预约”记录，显示成功提示。
8. 用户进入 U-P05，看到刚创建的预约。

### 8.2 预约失败流程

1. 用户在 U-P04 提交预约。
2. 服务器依次检查：当前体验用户是否有效 → 日期是否有效 → 时段是否可预约 → 座位是否启用 → 座位是否冲突 → 用户是否冲突。
3. 任一检查失败时停止写入，返回对应错误提示，并保留用户已选择的日期。
4. 用户修改日期或时段后可重新提交。

### 8.3 用户取消流程

1. 用户进入 U-P05，系统只查询当前 Session 用户的预约。
2. 对“已预约”且尚未开始的记录显示取消按钮。
3. 用户确认取消，服务器再次校验预约归属、存储状态和开始时间。
4. 校验通过后将存储状态改为“已取消”，记录取消时间和取消方为学生。
5. 页面显示取消成功，对应座位时段重新参与预约。

### 8.4 管理员管理预约流程

1. 管理员进入 A-P01，使用预置账号登录。
2. 登录成功后进入 A-P04，并可跳转到 A-P02。
3. 管理员在 A-P02 按日期或展示状态筛选预约。
4. 管理员对尚未结束的有效预约执行取消。
5. 服务器校验记录仍可取消后更新状态、取消时间和取消方，页面显示处理结果。

### 8.5 管理员维护座位流程

1. 管理员从 A-P04 进入 A-P03。
2. 新增时在 A-P03 打开同页新增状态，填写唯一座位编号、区域、备注和启停状态后保存。
3. 编辑时在 A-P03 打开同页编辑状态，修改允许字段并保存。
4. 停用座位后，U-P02 仍可展示该座位，U-P03 不提供预约入口，U-P04 服务端拒绝新预约。
5. 停用前已有预约不自动取消，由管理员在 A-P02 查看并按需要处理。

| 字段 | 含义 | 规则 |
| --- | --- | --- |
| Id | 预约编号 | 主键 |

| ReservationDate | 预约日期 | 只保存日期 |
| TimeSlot | 固定时段 | Morning、Afternoon、Evening 三选一 |
| StoredStatus | 存储状态 | Reserved、Cancelled 二选一 |
| CreatedAt | 创建时间 | 系统生成 |
| CancelledAt | 取消时间 | 未取消时为空 |
| CancelledBy | 取消方 | 未取消时为空；取消后为 Student 或 Admin |

### 11.5 对象关系与数据规则

1. 一个体验用户可以有多条预约，一条预约只属于一个体验用户。
2. 一个座位可以有多条不同日期或时段的预约，一条预约只对应一个座位。
3. 同一座位、同一日期、同一时段只能存在一条有效预约。
4. 同一体验用户、同一日期、同一时段只能存在一条有效预约。
5. 冲突判断只把 `StoredStatus = Reserved` 的记录视为有效记录。
6. 预约和座位均不做物理删除，保证历史数据可查看。

### 11.6 种子数据

1. 预置 2 个启用的体验用户。
2. 预置 1 个启用的管理员，README 提供课堂演示账号。
3. 预置 8 个座位，至少包含 2 个区域，其中 7 个启用、1 个停用。
4. 使用初始化器按首次建库当天生成 3 条相对日期预约：1 条未来有效预约、1 条已取消预约、1 条结束时间已过但存储状态仍为 Reserved 的预约，分别用于演示冲突、取消和动态“已完成”状态。

## 12. 状态流转说明

### 12.1 预约状态

数据库只保存两种状态，页面展示三种状态。

| 当前存储状态 | 触发条件 | 新存储状态 | 页面展示状态 | 是否终态 |
| --- | --- | --- | --- | --- |
| 无记录 | 合法预约提交成功 | Reserved | 已预约 | 否 |
| Reserved | 学生在开始前取消 | Cancelled | 已取消 | 是 |
| Reserved | 管理员在结束前取消 | Cancelled | 已取消 | 是 |
| Reserved | 当前时间大于或等于时段结束时间 | 不变 | 已完成 | 是（展示层） |
| Cancelled | 时间经过或重复操作 | 不变 | 已取消 | 是 |

展示状态计算规则：

1. `StoredStatus = Cancelled` 时始终展示“已取消”。
2. `StoredStatus = Reserved` 且当前时间早于结束时间时展示“已预约”。
3. `StoredStatus = Reserved` 且当前时间大于或等于结束时间时展示“已完成”。
4. 不运行后台定时任务，不批量把 Reserved 改写为 Completed。

取消边界：

1. 学生只能在当前时间早于预约开始时间时取消。
2. 管理员可以在当前时间早于预约结束时间时取消，包括已经开始但尚未结束的预约。
3. 当前时间大于或等于结束时间后，任何人都不能取消。

### 12.2 座位状态

| 当前状态 | 管理员操作 | 新状态 | 对新预约的影响 | 对已有预约的影响 |
| --- | --- | --- | --- | --- |
| 启用 | 停用 | 停用 | 禁止新建预约 | 保留，不自动取消 |
| 停用 | 启用 | 启用 | 恢复参与预约 | 原有记录不变化 |

座位启用不等于具体时段空闲。时段是否可预约仍需结合日期、时段和有效预约动态判断。

## 15. 风险与假设

### 15.1 风险

| 风险 | 可能影响 | 应对方式 |
| --- | --- | --- |
| 开发时间不足 | 用户端或管理端流程不完整 | 按体验账号与数据基础、学生预约闭环、管理端、统计的顺序开发；最后处理装饰细节 |
| 页面数量失控 | 被脚手架页面和重复表单拖慢 | 严格使用 U-P01 至 U-P05、A-P01 至 A-P04；取消使用页内按钮，新增/编辑共享 A-P03 同页表单 |
| 状态实现不一致 | 页面、筛选和统计出现不同结果 | 统一使用 StoredStatus + 当前时间计算 DisplayStatus，不新增 Completed 存储值 |
| 冲突校验只做在前端 | 可通过手工请求写入无效数据 | 所有日期、座位、用户和冲突规则在服务端重新校验 |
| 数据库关系过重 | 迁移和查询调试困难 | 只保留 4 类核心对象，区域和时段不单独建表 |
| LocalDB 环境差异 | 老师电脑启动失败 | README 写明 .NET 8、LocalDB、连接字符串和初始化命令 |
| 种子预约日期失效 | 课堂演示时没有有效预约 | 首次初始化使用相对日期，并提供重建演示数据库步骤 |
| AI 生成代码不稳定 | 编译失败或同一规则多种写法 | 每完成一个功能立即编译和测试，共用状态计算与校验逻辑 |
| 体验账号越权 | 用户可通过参数操作他人数据 | 服务端从 Session 取当前用户，并验证预约归属 |

### 15.2 假设

1. 系统用于本地课堂演示，不承载真实高并发和真实个人信息。
2. 服务器本地日期和时间可作为预约判断基准，不处理跨时区。
3. 每个固定时段内不处理签到、提前离座和临时占用。
4. 每个座位同一时段只允许一个体验用户使用。
5. 管理员账号和体验用户由系统预置，不提供账号管理页面。
6. 区域数量和座位数量较少，单条件筛选即可满足演示。
7. 停用座位不会自动取消已有预约，由管理员人工处理。
