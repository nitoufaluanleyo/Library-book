## 4. 做得好的地方

1. 项目范围始终保持 5 个用户端页面和 4 个管理端页面，没有扩展支付、扫码等超纲功能。
2. 页面卡、静态原型和截图覆盖完整，用户端手机优先、管理端桌面优先的视觉方向清楚。
3. 数据库设计明确了冲突索引、状态、并发和统计口径，具备后续实现依据。
4. 任务板保留了 Sprint 0 至 Sprint 4 的任务、完成率和门禁。
5. 后续阶段没有在缺少工程时伪造 build、数据库、账号或测试通过记录。
6. README 终稿将“设计目标”和“实际交付”分开，老师可以快速判断仓库状态。

## 8. 如果再给 1 至 2 次课，最该补什么

### 第 1 次课：先得到可运行最小系统

1. 修正并复核 3 个开发前 P0。
2. 创建 .NET 8 MVC Solution 和 Web 项目。
3. 实现 4 个实体、DbContext、Migration 和幂等种子数据。
4. 完成首次 build、run 和 LocalDB 建库。
5. 至少打通体验用户切换、座位列表、预约写入和我的预约回看。

### 第 2 次课：补管理闭环和验收证据

1. 完成管理员哈希登录和 AdminCookie 保护。
2. 完成预约筛选、管理员取消、座位启停和用户端状态回流。
3. 补 Playwright msedge 两条主链路和 smoke 脚本。
4. 修复 P0，完成回归并更新 README 真实账号和命令。

如果只能再上一节课，应优先完成“可 build + 可建库 + 一次真实预约写入和回看”，而不是继续做样式。

## 9. 最终演示顺序建议

### 9.1 当前仓库可演示顺序

1. 打开 README，先说明当前是文档和静态原型成果，不是可运行系统。
2. 展示 docs/03、docs/04，说明固定范围和 9 页结构。
3. 打开 `prototype/static-v2/index.html`，按 U-P01 至 U-P05 浏览用户端。
4. 从管理员入口浏览 A-P01 至 A-P04。
5. 展示空状态、冲突提示和后台错误等静态状态。
6. 打开 `database/README.md`，说明 Code First 设计和未实现缺口。
7. 打开 docs/16，说明 5 个 P0、2 个 P1 和未执行测试。

演示时不能声称登录、预约、取消、统计或状态回流已经真实生效。

### 9.2 功能完成后的目标演示顺序

1. 重建数据库并启动应用。
2. 切换体验账号，查看座位并创建预约。
3. 在“我的预约”查看记录和冲突提示。
4. 管理员登录，筛选并取消该预约。
5. 返回用户端查看已取消和时段释放。
6. 展示座位启停和五项统计变化。
7. 执行 Playwright、smoke 和数据库复位命令。

该顺序是后续目标，不是当前可演示能力。

## 11. 复盘结论

本项目在产品、设计和原型阶段形成了较完整的过程材料，但开发门禁复核未通过后，后续阶段持续停留在阻塞记录，没有完成代码落地。最大的教训是：文档阶段结束后必须尽快建立一个最小可运行纵向切片，先验证 build、数据库和一次真实业务闭环，再继续扩展文档和体验。

当前最负责的交付方式不是美化完成度，而是明确告诉验收者：哪些材料真实存在、哪些功能只在原型中、哪些命令当前不可执行，以及下一次开发应该从哪里恢复。
